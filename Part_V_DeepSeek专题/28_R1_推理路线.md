## 摘要

OpenAI o1(2024.09)首次大规模演示"test-time reasoning":让模型在推理时输出超长思维链,benchmark 显著上升。这一路线长期闭源。**DeepSeek-R1(2025.01)以开源方式跑通了同一范式**,且暴露惊人事实:

- **R1-Zero**:在 V3-Base 上**直接跑 GRPO,完全跳过 SFT**,模型**自发涌现长思维链**与 "aha moment"
- **R1**:在 R1-Zero 经验上加 SFT + 拒绝采样 + 第二轮 RL + 通用 SFT 的 4 阶段管线,质量与 o1 相当
- **R1 蒸馏**:用 R1 的 reasoning 数据 SFT 小模型(Qwen-1.5B/7B/14B/32B、LLaMA-70B),把 reasoning 灌进小尺寸

读完你能解释 GRPO 在 R1 中的具体形式、为什么"纯 RL 也能涌现",以及如何用 R1 蒸馏数据强化你自己的小模型。

## 28.1 背景:从 GPT-o1 到开源 reasoning

### 28.1.1 o1 揭示的新维度

2024.09 OpenAI 发布 o1-preview。和 GPT-4 比,**它在数学、编程、PhD 级科学题**上跳跃式提升。关键变化:

- 模型推理时输出**很长的思维链**(隐式 CoT)
- API 不返回中间过程,只返回最终答案
- 收费按"reasoning token + output token"双重计

提出新概念:**test-time scaling**——把推理 token 数(思考时长)也当成 scaling 维度(§17.8)。

### 28.1.2 开源圈的难题

社区无法直接学 o1,只看到现象,不知道训练方法。各种推测:

- **MCTS / process reward**(过程奖励)
- **SFT 海量人工 CoT 数据**
- 全新架构

→ R1 论文给出意外答案:**纯 RL + 简单可验证奖励就能涌现长思维链**,远比想象简单。

## 28.2 R1-Zero:纯 RL 涌现 reasoning

### 28.2.1 流程极简

```text
1. base = V3-Base(预训练完,未 SFT 未对齐)
2. 直接跑 GRPO:
     prompt = 数学题 / 代码题
     reward = 答案对 + 格式对
3. 训练直到 reasoning 涌现
```

**完全跳过 SFT**,模型从未见过"演示 CoT 数据",仍然学会写思维链。

### 28.2.2 涌现的三件神事

R1-Zero 训练曲线观察到:

1. **思维链长度自然增长**:从 ~ 500 token 增长到 5000-10000 token,无人为干预
2. **"Aha moment"**:模型自发出现 "Wait, let me reconsider" / "Actually, that's wrong" 等自我修正
3. **reasoning 准确率随推理 token 数提升**:test-time scaling 实证

### 28.2.3 R1-Zero 的问题

虽然能力惊人,但**输出风格糟糕**:

- 中英混杂、格式不稳
- 难读、过冗长
- 不易接业务

→ 引出"完整版 R1"。

## 28.3 R1 的多阶段管线

R1 不是 R1-Zero 的简单替代,而是用一个**多阶段管线**把 R1-Zero 的能力"驯化"成可用模型:

```text
Stage 1: Cold-Start SFT
    数据 = 用 R1-Zero 生成大量 reasoning trace,人工/规则筛选成"高质量 cold-start" 数据
    在 V3-Base 上 SFT,作为良好起点

Stage 2: Reasoning RL(GRPO)
    奖励 = 答案正确 + 格式 + 语言一致性(防中英混杂)
    生成大量 reasoning 数据(用作下一阶段)

Stage 3: Rejection Sampling + SFT
    从 Stage 2 模型多次采样,选出最优 trace
    + 通用任务数据(对话 / 写作 / 工具),让模型不仅能 reasoning,也能日常对话
    SFT 一次

Stage 4: Final RL
    再次 GRPO,reward 加入对齐(helpful + harmless)
    最终模型 = DeepSeek-R1
```

整个管线核心思想:**用 RL 涌现能力,用 SFT 驯化风格 + 加上通用能力**。

### 28.3.1 与传统 RLHF 的不同

传统 RLHF 主要做风格对齐,reward 是人类偏好。
R1 把 RL 用在**任务正确性**上(数学答案对、代码通过测试)→ **可验证奖励**,信号干净。

## 28.4 GRPO 在 R1 中的具体实现

(机制详见 §16.7)

### 28.4.1 采样

对每个 prompt $x$,**采样 $G$ 个完整回答** $\{y_1, \ldots, y_G\}$。

### 28.4.2 奖励 + 优势

每个回答得一个 scalar reward $r_i$,组内归一化得 advantage:

$$
A_i = \frac{r_i - \text{mean}(\{r_j\})}{\text{std}(\{r_j\}) + \epsilon} \tag{28.1}
$$

### 28.4.3 PPO 风格更新(无 value model)

$$
\rho_i = \frac{\pi_\theta(y_i|x)}{\pi_{\theta_\text{old}}(y_i|x)}
$$

$$
\mathcal{L}_{\text{GRPO}} = -\frac{1}{G}\sum_i \min\!\big(\rho_i A_i,\, \text{clip}(\rho_i, 1\!-\!\epsilon, 1\!+\!\epsilon)\, A_i\big) + \beta \cdot \text{KL}(\pi_\theta \| \pi_{\text{ref}}) \tag{28.2}
$$

KL 锚是 SFT 模型(或 base)。$\beta$ 不大(0.01-0.05),让模型在 reasoning 上自由探索。

### 28.4.4 优势

- 无 value model → 显存少 1 个大模型
- 无 reward model(用规则验证)→ 显存少 1 个
- group baseline 比 single-sample baseline 方差更小
- 支持可验证 reward,不污染由 RM 引入的偏差

R1 之后,GRPO 几乎成为"reasoning RL"的开源默认范式,Qwen / Llama 等团队都已采用。

## 28.5 奖励函数设计

R1 的 reward 是**简单 + 可验证**的组合:

### 28.5.1 准确性奖励

- 数学题:答案匹配标准答案(支持表达式等价)
- 代码题:运行单测,全过给正分
- 多选题:选对得分

### 28.5.2 格式奖励

强制输出格式:

```text
<think>
... 思考过程 ...
</think>

<answer>
最终答案
</answer>
```

未按格式输出扣分。这给"思维链显式化"提供了重要 inductive bias。

### 28.5.3 语言一致性(R1 阶段加入)

R1-Zero 中文 + 英文混杂严重。R1 训练时加 **语言奖励**:回答语言与提问一致才得分。

### 28.5.4 不引入 RM

整个 R1 的 RL 阶段**没有偏好 RM** —— 全部是规则可验证。这避免了 reward hacking 的最大风险源(§16.9.3)。

## 28.6 涌现现象:思维链自然变长 / aha-moment

### 28.6.1 长度演化曲线

R1 论文 Figure 3:

- 训练初期:平均输出 ~500 token
- 训练中期:~2000 token
- 训练后期:~5000-10000 token

**模型自己学到"想得久 → 答得更准"**,没有人为干预长度。

### 28.6.2 自我修正

随着训练,模型开始出现:

- "Wait, that doesn't seem right..."
- "Let me reconsider step 3..."
- "Actually, going back to the original equation..."

→ 自发的 self-correction。这不是 SFT 教的,是 RL 在追逐 reward 时**自然演化**出的策略。

### 28.6.3 为什么会涌现?

直觉解释:

1. base 模型本来就**有 latent reasoning 能力**(预训练数据里有大量数学 / 代码 / 论证)
2. RL 的 reward 把"答对"这条路径的概率拉高
3. 答对的最稳路径就是"边想边算 + 自检",**这恰好对应长思维链**
4. 长思维链的 token 数本来就让 reward 期望上升 → 模型自然偏向更长输出

更严格的 mechanistic interpretability 工作还在进行中(2025+ 热门方向)。

## 28.7 蒸馏:把 reasoning 灌进 7B/14B/70B

### 28.7.1 蒸馏数据

把 R1 生成的 reasoning trace(数学、代码、科学题)收集起来,组成 (prompt, reasoning, answer) 数据集,DeepSeek 公开了 80 万条 R1 蒸馏样本。

### 28.7.2 蒸馏方法

**只用 SFT,不做 RL**(节省成本):

- 学生模型:Qwen-1.5B / 7B / 14B / 32B / LLaMA-8B / LLaMA-70B
- 标准 NTP loss,与 §15 SFT 完全相同
- 训练几个 epoch

### 28.7.3 惊人结果

| 模型 | 蒸馏后 MATH | 备注 |
| --- | --- | --- |
| Qwen-7B | 80+ | 超过 GPT-4o 原版 |
| LLaMA-70B | 90+ | 接近 GPT-4 reasoning |
| Qwen-32B | 90+ | 与 o1-mini 相当 |

→ **reasoning 能力可被"用数据"压缩进小模型**。这与 §17.5 涌现能力的讨论形成有趣呼应。

### 28.7.4 工程意义

- 单卡 / 单机即可跑出强 reasoning(小模型 + R1 蒸馏)
- 蒸馏数据成本几乎为零(开源)
- 商业落地友好

→ 2025 开源圈的"R1 蒸馏潮"由此而生。

## 28.8 Test-Time Scaling 的实证

### 28.8.1 经验法则

R1 系列证明:

- 推理 token 数 ×2 → accuracy 大约 +几个百分点
- 这是 log-linear scaling,不会饱和(在常见 benchmark 上)

→ "**多想一会儿,就更准**"成了开源验证的 scaling law。

### 28.8.2 工程含义

- 同模型,**提供更长 reasoning budget,质量上升**
- "thinking 长度"成为新的 scaling 维度,与 model size / data 并列(§17.8)
- 在 API 层,可让用户选"思考预算"

### 28.8.3 局限

- 长 reasoning ≠ 总是更准:简单问题可能 overthink
- token 成本与 latency 显著上升
- 还在探索"自动决定 think 长度"(动态预算)

## 28.9 工程要点与训练账

### 28.9.1 R1 训练规模

- 基座:V3-Base(671B 总,37B 激活)
- RL 阶段 token 总量:数千亿级(包括多次采样)
- 公开成本:未明确,但估计**远小于 V3 预训练**(几 M USD 量级)

### 28.9.2 RL 工程难点

- **超长输出**(数千 token):batch 显存膨胀,需要长上下文 attention + chunked
- **高方差 reward**:可验证 reward 在错误答案上是 hard 0,梯度噪声大
- **KL 漂移**:不加 KL 锚会让模型立刻 mode collapse
- **过长 trace 的 ROUGE 不可比**:回到 final answer 才打分

### 28.9.3 监控

- 平均输出长度(看是否涌现)
- accuracy(主目标)
- KL(确保不漂离 base)
- format reward(确保格式)
- 语言一致性(R1 阶段)

## 28.10 完整伪代码:R1-style GRPO

```python
# 伪代码,实际用 trl + 自家训练框架
# 单个 step:
# 1. 用 πθ_old 对 prompt 采样 G 个回答
# 2. 计算每个回答的 reward(可验证 + 格式 + 语言)
# 3. 组内归一化得 advantage
# 4. PPO 风格更新 πθ
# 5. 加 KL(πθ || π_ref)正则

def r1_grpo_step(prompt, policy, ref_policy, optimizer,
                 G=16, beta=0.03, eps=0.2):
    # 1. 采样 G 个回答(从 πθ_old)
    samples = []
    for _ in range(G):
        y = policy.generate(prompt, max_new_tokens=8000, temperature=1.0)
        samples.append(y)
    # 2. 计算 reward
    rewards = [reward_fn(prompt, y) for y in samples]
    # 3. 归一化 advantage
    r = torch.tensor(rewards, dtype=torch.float)
    A = (r - r.mean()) / (r.std() + 1e-6)

    # 4. 计算新策略 log-probs(在所有 sample 上)
    total_loss = 0.0
    for y, adv in zip(samples, A):
        # 与 §16.10 类似:对 y 的 token 级 log-prob 求和
        new_lp = policy.log_prob(prompt, y)
        old_lp = old_policy.log_prob(prompt, y).detach()    # 上一步 snapshot
        ratio = (new_lp - old_lp).exp()

        # clipped surrogate
        unclipped = ratio * adv
        clipped = ratio.clamp(1 - eps, 1 + eps) * adv
        ppo_loss = -torch.min(unclipped, clipped).mean()

        # KL 锚
        ref_lp = ref_policy.log_prob(prompt, y).detach()
        kl = (new_lp - ref_lp).mean()

        total_loss = total_loss + ppo_loss + beta * kl

    total_loss = total_loss / G
    total_loss.backward()
    optimizer.step()
    optimizer.zero_grad()
    return total_loss.item()


def reward_fn(prompt, y):
    answer = extract_answer(y)
    score = 0.0
    if check_correctness(prompt, answer): score += 1.0
    if check_format(y):                   score += 0.2
    if check_language(prompt, y):         score += 0.1
    return score
```

> 教学版,生产实现需要分布式、long-context 优化、ZeRO-3 等。社区已有不少 R1 复现项目(open-r1、verl 等)。

## 🎨 28.10.A 通俗类比合集

### 类比一: 学生自学解题 📚

> R1-Zero = 把学生关进图书馆,只给他"答案正确加 1 分"的反馈,**不给任何示范**。

- 学生最初瞎写,大多错
- 慢慢发现"边写边验算"得分高 → **自发学会写演算过程**
- 过段时间:写下"嗯…等等,我刚才那一步好像错了" → **自发学会自检**
- 长卷子总是好过短卷子 → 卷子越写越长

R1 = 上完一段图书馆课后,再给他几堂"作文风格课"和"待人接物课"(SFT + RL),让他既能解题又能写得好看。

### 类比二: 武林秘籍自学 🥋

> R1-Zero 像独自闭关。

- 没有师父示范(无 SFT)
- 只有一面墙(reward),打中加分
- 几个月后练出"自言自语 + 自我纠正"的怪招 → 招式厉害但风格野
- R1 = 出关后由师门规范一下招式(SFT 风格 + 对齐 RL)
- 蒸馏 = 把秘籍部分招式抄给师弟(小模型)

### 类比三: 健身教练 vs 自练 💪

> 传统 RLHF = 有教练 RM 全程盯;R1 = 没教练,只看体测仪(可验证奖励)。

- **体测仪 reward**:做对一组动作仪器叫 ✓,做错叫 ✗
- 没人指导动作姿势,但**为了"对",自然涌现出**热身 + 调息 + 自检
- 长锻炼时长(think 长)对应更稳定结果
- 蒸馏 = 把你的训练日志给同伴照搬,他不用闭关也能进步

### 三个类比对比

| 类比 | 强调什么 | 适合记忆什么 |
| --- | --- | --- |
| 📚 自学 | **无示范涌现** | R1-Zero 的奇观 |
| 🥋 闭关 | **野招 → 规范** | R1 多阶段管线 |
| 💪 健身 | **可验证 reward** | GRPO 的奖励设计 |

**记忆口诀**:
> **"R1-Zero 闭关纯 RL,R1 多阶段驯化,蒸馏装进小模型;答对加分 + 自检涌现,想得久就答得准。"**

## 28.11 常见疑问

### Q1: R1-Zero 真的没用 SFT 吗?
**A**: 完全没用。直接在 V3-Base(预训练完,未 SFT)上跑 GRPO。论文反复验证。

### Q2: 为什么纯 RL 没让模型"输出乱码"?
**A**: 因为 V3-Base 本身已经会写连贯文字(预训练目标),RL 只是**重新加权**已有的语言能力,把"会答对"的分布拉上来。加上 format reward + KL 锚的护栏,稳定。

### Q3: R1-Zero 模型可以直接发布吗?
**A**: 论文公开了,但因为风格糟糕(中英混杂、过冗长)不适合直接用作 chat。生产用 R1 + 蒸馏版。

### Q4: 我能复现 R1 训练吗?
**A**: 大模型(671B base + 长 RL)难;但**蒸馏版可复现**——用开源 R1 蒸馏数据 SFT 一个 7B 模型,几张 A100 就够。

### Q5: GRPO 与 PPO 谁更好?
**A**: 对**可验证 reward**(数学 / 代码)场景,GRPO 性价比更高,质量也不差;对**偏好 RM** 场景,PPO 仍有优势(value 估算更准)。R1 选 GRPO 是任务驱动。

## 28.12 本章小结

### 核心管线

```text
R1-Zero:  V3-Base → 纯 GRPO(可验证 reward) → reasoning 涌现
R1:       V3-Base → cold-start SFT → GRPO → rejection sampling SFT → final RL
蒸馏:     R1 生成 reasoning data → SFT 小模型(1.5B-70B)
```

### 核心要点

| 要点 | 关键 |
| --- | --- |
| **R1-Zero** | 纯 RL 涌现长思维链 |
| **GRPO** | group baseline,无 value,可验证 reward |
| **奖励设计** | 准确性 + 格式 + 语言一致 |
| **多阶段管线** | SFT-RL-SFT-RL,驯化风格 + 加通用能力 |
| **蒸馏** | reasoning 能力可压进小模型 |
| **Test-time scaling** | 思考越久越准 |

### 一句话总结

> **R1 路线在开源世界跑通了 o1 同款范式:用 V3-Base 直接做 GRPO + 可验证奖励,模型自发涌现长思维链与自我修正;经多阶段管线驯化,质量与 o1 相当;再把 reasoning 蒸馏到 7B-70B 小模型,让全社区都能用——这是 LLM 史上第一次清楚展示"RL + 简单奖励" 能解锁全新能力维度的开源实证。**

## 28.14 实战练习

### 练习 1 (★)
