## 摘要

LLM 训练已经溢出**单卡、单机、单数据中心**:7B+ 模型连权重都装不进 80 GB,70B+ 需要数百卡,671B MoE 需要数千卡。本章系统梳理五种并行维度及其组合:

- **DP / DDP**:数据并行,每卡完整模型 + 不同数据
- **ZeRO-1/2/3**:把优化器状态 / 梯度 / 权重切片
- **TP**:矩阵切列切行,跨卡做单个矩阵乘
- **PP**:把模型按层切给不同 stage,流水线推进
- **SP**:激活按序列维度切,减少 TP 的激活显存
- **EP**:把 MoE 专家切给不同卡,All-to-All 路由
- **DualPipe**(DeepSeek-V3):前向 / 反向交错的高级 PP

每种维度都有自己的通信代价与适用 batch 大小;真实系统是 **3D / 4D / 5D 并行的组合**。

## 14.1 为什么单卡训不动 LLM?

回顾 §04.5.5 / §13.6.2,训练显存包含:

| 项 | 大小(bf16 + AdamW) |
| --- | --- |
| 权重 | 2P |
| 梯度 | 2P |
| Adam m, v | 8P |
| fp32 master θ | 4P |
| 激活 | 与 B、N、T、d 成正比,常常是上述之和 |
| **总计** | ≈ **16P + 激活** |

**7B 模型**:训练显存 ≈ 112 GB → 单 H100 80GB 装不下,**必须** ≥ 2 卡。
**70B 模型**:训练显存 ≈ 1120 GB → 必须 ≥ 16 卡。
**DeepSeek-V3 671B**:仅权重 + 优化器状态 ≈ 11 TB → 必须 ≥ 数百卡。

> 这就是为什么"分布式"不是高级话题,而是 LLM 训练的**入门门槛**。

## 14.2 五种并行维度总览

| 维度 | 切什么 | 通信代价 | 适合规模 |
| --- | --- | --- | --- |
| **DP / DDP** | 数据 | All-Reduce 梯度 | 任意 |
| **ZeRO-1/2/3** | 优化器 / 梯度 / 权重 | Reduce-Scatter + All-Gather | 中-大模型 |
| **TP** | 单层内矩阵列 / 行 | All-Reduce 中间激活(每层) | 单机内多卡(高带宽) |
| **PP** | 跨层切 | Stage 间点对点 | 跨节点 |
| **SP** | 序列长度 | All-Gather 激活 | 与 TP 联用 |
| **EP** | MoE 专家 | All-to-All token 路由 | MoE 模型 |

LLM 大规模训练几乎都是这些维度的**多维笛卡尔积**。

## 14.3 Data Parallel(DP / DDP)

### 14.3.1 思路

每卡持完整模型,把 batch 拆成 N 份 micro-batch 给 N 卡。

- forward / backward 各自独立
- backward 完成后**全卡 all-reduce 梯度**(每个参数的梯度做求平均)
- 每卡用同样的梯度走 optimizer.step → 模型保持完全一致

### 14.3.2 DataParallel vs DistributedDataParallel(DDP)

- `nn.DataParallel`:单进程多 GPU,**python GIL 瓶颈**,几乎不用
- `nn.parallel.DistributedDataParallel`(DDP):多进程,每进程一卡,**生产标准**

PyTorch `torchrun` 启动多进程,各进程通过 NCCL 通信。

### 14.3.3 通信代价

一次 step 的梯度 all-reduce 量 = $2P$ bytes(bf16),用环形 ring-allreduce 实际通信约 $2P(N-1)/N \approx 2P$。

随 $P$ 增大,通信成本会成为瓶颈 → 与 ZeRO 联合优化。

### 14.3.4 与 gradient accumulation 配合

DDP 默认每次 backward 都 all-reduce → 多 micro-batch 累积时浪费。用:

```python
with model.no_sync():
    # 前 N-1 个 micro 不同步
    loss.backward()
# 最后一个 micro
loss.backward()
optimizer.step()
```

## 14.4 ZeRO:把优化器状态切片

### 14.4.1 动机

DDP 下每卡都存完整 16P bytes,**90% 是重复的**。ZeRO(Rajbhandari et al., 2020)把这些重复分摊到 N 卡。

### 14.4.2 三档切分

| 等级 | 切什么 | 单卡显存 | 通信代价 |
| --- | --- | --- | --- |
| **ZeRO-1** | 优化器状态(m, v, fp32 master) | (4P + 2P + 2P + 12P/N) | 与 DDP 相同 |
| **ZeRO-2** | + 梯度 | (4P + 12P/N + 2P/N) | + 略 |
| **ZeRO-3** | + 权重 | (16P / N) | 显著增加(每层 forward 都要 all-gather 权重) |

实际工程中:

- 7B-13B 用 ZeRO-1 / 2
- 70B+ 用 ZeRO-3 或 TP + PP
- 训练通信高的话再加 offload 到 CPU / NVMe(ZeRO-Infinity)

### 14.4.3 ZeRO-3 forward 流程

```text
for each layer:
    1. all-gather 该层的权重碎片到一张完整 W
    2. forward 计算
    3. 释放完整 W,只保留碎片
backward 同理
```

显存最低、通信最高,**带宽足够时是 70B 训练的主力**。

### 14.4.4 PyTorch FSDP

FSDP(Fully Sharded Data Parallel)是 ZeRO-3 的 PyTorch 原生实现,与 DDP 同 API,广泛使用:

```python
from torch.distributed.fsdp import FullyShardedDataParallel as FSDP
model = FSDP(model)
```

## 14.5 Tensor Parallel(TP):矩阵切列切行

### 14.5.1 动机

单个 Linear $Y = X W$ 太大装不下一张卡时,把 $W$ **横切或纵切**给多卡。

### 14.5.2 列切 + 行切(Megatron-LM 风格)

考虑 FFN 的两层:

```text
h = SwiGLU( W1 x )       # W1: (d, d_ff) — 列切
y = W2 h                 # W2: (d_ff, d) — 行切
```

**列切 W1**:把 $W_1$ 按列均分到 N 卡,每卡得 $(d, d_{\text{ff}}/N)$。每卡独立算自己那段 $h_i$,**无通信**。

**行切 W2**:把 $W_2$ 按行均分,每卡持 $(d_{\text{ff}}/N, d)$,与对应的 $h_i$ 相乘得 $y_i$,**最后 all-reduce 求和** $y = \sum y_i$。

```text
y = AllReduce( sum_i W2_i @ h_i )
```

每个 Block 内 Attention 与 FFN 各做一次 all-reduce → **每层 2 次** all-reduce,**通信很重**,只能在单机内 NVLink/NVSwitch(900 GB/s+)使用。

### 14.5.3 Attention 的 TP

把多 head 分给不同卡:每卡负责 $h/N$ 个 head,Q/K/V 投影列切,O 投影行切,与 FFN 同构。

### 14.5.4 缺点

- 通信频繁(每层 2 次 all-reduce)
- 受单机内 GPU 数限制(典型 8 卡)
- 不适合跨节点(网络带宽差几个数量级)

## 14.6 Pipeline Parallel(PP)与 1F1B / DualPipe

### 14.6.1 思路

把 N 层 Transformer 切成 K 个 **stage**,每 stage 在一组卡上;输入按"流水线"流过各 stage。

### 14.6.2 朴素 PP 的"气泡"问题

```text
GPU0: f1 f2 f3 ... 等待回流 ... b3 b2 b1
GPU1:    f1 f2 f3 ... b3 b2 b1
GPU2:       f1 f2 f3 ... b3 b2 b1
...
```

每张卡有大量"等数据"空闲时间,称为 **pipeline bubble**,与"micro-batch 数 / stage 数"成反比。

### 14.6.3 1F1B(One Forward One Backward,Megatron 风格)

把整个 batch 拆成 K 个 micro-batch:

- 前 K 步:每 stage 各做 1 个 forward(预热)
- 中间:每 stage 交替 1F1B(forward 一个 + backward 一个)
- 最后 K 步:每 stage 各做 1 个 backward(冷却)

**气泡比** ≈ $(K - 1) / (m + K - 1)$,$m$ = micro-batch 数。

### 14.6.4 DualPipe(DeepSeek-V3)

DualPipe 把 forward 与 backward **不同方向同时流动**(双向流水线),并把 compute / 通信重叠:

- 一组 micro-batch 从左到右(forward)
- 另一组从右到左(backward)
- 通信(All-to-All for MoE 专家)与计算重叠

实测把气泡比降到 ~5%,显著优于 1F1B,是 DeepSeek-V3 训出 671B / 14.8T token / $5.5M 成本的关键工程基础。

### 14.6.5 跨节点 vs 跨机内

PP 的 stage 间通信是 **点对点**(每 micro-batch 只传一次激活),远小于 TP。**适合跨节点**,典型部署:

- 单机内 8 卡 → TP
- 跨机 → PP

## 14.7 Sequence Parallel(SP)

### 14.7.1 动机

TP 切了 W 的列 / 行,但 **激活仍按 (B, T, d) 完整保留**——长 T 时激活显存爆炸。

SP 把激活按 **序列维度 T** 切给 N 卡:每卡只持 $T / N$ 个 token 的激活。

### 14.7.2 与 TP 配合

在 Norm / Dropout 等 "逐 token 独立" 的层上做 SP(序列切),进入 Attention / FFN 前 All-Gather 激活到完整序列,再做 TP。

Megatron-LM SP 论文:在 TP=8、序列 8k 时,激活显存可降 60%。

### 14.7.3 与 Ring Attention / Tree Attention

对超长序列(> 100k),Attention 自身也按序列切(Ring Attention),让 KV cache 分布到多卡。这是 § 21 推理优化的话题,但训练时也有类似技术。

---

## 14.8 Expert Parallel(EP)

### 14.8.1 动机

MoE 有 N 个专家,但每个 token 只激活 Top-k。把不同专家放在不同卡上 → **总参数加 N 倍而单卡显存不变**。

### 14.8.2 通信:All-to-All

```text
1. Router 决定每个 token 去哪些专家
2. 把 token 按目标专家分组(All-to-All:把"我手上想去专家 j 的 token"发给持有专家 j 的卡)
3. 每张卡独立跑自己持有的专家
4. 再 All-to-All:把计算结果发回原 token 所属卡
```

**All-to-All 通信是 MoE 的最大瓶颈**,DeepSeek 投入了大量工程来优化(DualPipe / 拓扑感知路由)。

### 14.8.3 与负载均衡

MoE 训练需要辅助 loss 让 token 平均分配到各专家,否则某些卡总在等(详见 §23 DeepSeekMoE)。

## 14.9 多维并行的组合:3D / 4D / 5D

实际配置一般是多维笛卡尔积:

$$
\text{TotalGPUs} = N_{\text{DP}} \times N_{\text{TP}} \times N_{\text{PP}} \times N_{\text{EP}} \times \ldots
$$

例:DeepSeek-V3 训练用 ≈ 2048 GPU,可能配置:

- DP=4(数据并行)
- PP=16(跨节点)
- TP=1(无 TP,因为 MLA 单卡放得下)
- EP=64(专家并行,256 专家 / 64 = 4 专家/卡)

DeepSeek 自研 DualPipe + 拓扑感知 All-to-All,让 EP / PP 通信几乎全重叠到 compute 后。

---

## 14.10 通信原语:All-Reduce / All-Gather / Reduce-Scatter / All-to-All

| 原语 | 含义 | LLM 用处 |
| --- | --- | --- |
| **All-Reduce** | N 张卡每张持一份数据 $x_i$,全卡得到 $\sum x_i$ | DDP 梯度同步、TP 行切后求和 |
| **All-Gather** | N 张卡各持 $x_i$,全卡得到拼接 $[x_1, ..., x_N]$ | ZeRO-3 取权重、SP 还原序列 |
| **Reduce-Scatter** | 全卡有完整 $x$,reduce 后**分发**各自一段 | ZeRO 切梯度;All-Reduce 的"前半" |
| **All-to-All** | 每张卡有 N 段数据,各发给对应卡,全卡互换 | MoE EP 的 token 路由 |
| **Broadcast** | 单卡到全卡 | 初始权重分发 |

通信代价:All-Reduce ≈ 2x;All-Gather + Reduce-Scatter = All-Reduce(等价但更细粒度)。

## 14.11 完整代码:最小 DDP + ZeRO-2 demo

```python
# train_ddp.py — 用 torchrun 启动:
# torchrun --standalone --nproc_per_node=2 train_ddp.py
import os, torch
import torch.distributed as dist
import torch.nn as nn
from torch.nn.parallel import DistributedDataParallel as DDP

def setup():
    dist.init_process_group("nccl")
    rank = int(os.environ["LOCAL_RANK"])
    torch.cuda.set_device(rank)
    return rank

def cleanup():
    dist.destroy_process_group()

def main():
    rank = setup()
    device = torch.device(f"cuda:{rank}")

    # 1. 构建模型
    model = nn.Sequential(nn.Linear(1024, 4096), nn.GELU(),
                          nn.Linear(4096, 1024)).to(device)
    model = DDP(model, device_ids=[rank])

    # 2. ZeRO-2 优化器(切优化器 + 梯度)
    from torch.distributed.optim import ZeroRedundancyOptimizer
    optimizer = ZeroRedundancyOptimizer(
        model.parameters(), optimizer_class=torch.optim.AdamW,
        lr=3e-4, betas=(0.9, 0.95))

    # 3. 训练循环
    for step in range(100):
        x = torch.randn(8, 1024, device=device)
        y = torch.randn(8, 1024, device=device)
        loss = ((model(x) - y) ** 2).mean()
        loss.backward()
        optimizer.step()
        optimizer.zero_grad(set_to_none=True)
        if rank == 0 and step % 10 == 0:
            print(f"step={step} loss={loss.item():.4f}")

    cleanup()

if __name__ == "__main__":
    main()
```

进阶版用 FSDP 即可达到 ZeRO-3,API 极为相似。

## 🎨 14.12.A 通俗类比合集

### 类比一: 餐厅多人协作 🍽️

> 模型 = 一道菜,GPU 卡 = 厨师。

- **DP** = 每个厨师都做一道完整菜,各自服务一桌客人(数据并行)
- **TP** = 切菜分给多个人,一个切丝、一个剁泥,合作做同一道菜
- **PP** = 流水线:前菜师傅 → 主菜师傅 → 甜品师傅,接力完成每一桌
- **EP** = 川菜、粤菜、湘菜师傅同坐一店,每桌客人按口味分流到不同师傅(MoE)
- **ZeRO** = 调料柜共享:每个厨师不必都准备整套调料,各负责一部分,要用时去拿
- **SP** = 一道菜按"前 1/3、中 1/3、后 1/3"分给三人同时炒(序列分片)

### 类比二: 工厂生产线 🏭

> 训练 = 生产线上每天造 N 台机器。

- **DP** = 多条平行生产线,各自造一台完整机器
- **TP** = 一台机器太大,机身切成几段,几条线各做一段
- **PP** = 一条长生产线:工位 A → B → C → D 接力完成
- **EP** = 多种型号同厂生产,不同型号去不同车间
- **DualPipe** = 流水线双向跑:产品从左到右,质检报告从右到左,同时不空闲
- **All-Reduce** = 每条线产出 1000 个零件,最后中央仓汇总

### 类比三: 学校批改试卷 📝

> 数千份试卷要改完。

- **DP** = N 个老师各改 N 份不同试卷,最后汇总分数(all-reduce)
- **TP** = 一份试卷太长,5 位老师每人改一道大题,合起来才是完整成绩
- **PP** = 流水线批改:语文老师 → 数学老师 → 英语老师传递同一份试卷
- **ZeRO-3** = 标准答案文件太大,每位老师只持有其中一段,需要时互相借
- **EP** = 学校 256 位特长老师(MoE 专家),每份试卷只给 2 位最对路的看

### 三个类比对比

| 类比 | 强调什么 | 适合记忆什么 |
| --- | --- | --- |
| 🍽️ 餐厅 | **协作方式** | 五种并行各自风格 |
| 🏭 工厂 | **节奏与吞吐** | PP / DualPipe |
| 📝 改卷 | **资源分摊** | ZeRO / EP / TP |

**记忆口诀**:
> **"DP 拆数据,TP 拆矩阵,PP 拆层数,EP 拆专家,SP 拆序列;ZeRO 把重复砍掉。"**

## 14.13 常见疑问

### Q1: 我训 7B 模型,选 ZeRO-2 还是 ZeRO-3?
**A**: 优先 ZeRO-2(或 ZeRO-1 + offload)。ZeRO-3 通信更重,在 8 卡内带宽不足时反而变慢。70B+ 才需要 ZeRO-3 / FSDP。

### Q2: TP=8 与 ZeRO-3 是不是重复?
**A**: 不重复,正交。TP 切单层内,ZeRO-3 切跨层副本。大模型常 TP=8 内部 + ZeRO-3 外部组合用。但 DeepSeek-V3 因 MLA 把激活 / KV 大大压缩,不需要 TP。

### Q3: PP 的 micro-batch 数怎么选?
**A**: 经验:$m \geq 4 \times K$($K$ = stage 数)。$m$ 越大气泡越小,但占显存(激活要存 $m$ 份)。

### Q4: 千卡训练频繁出错怎么办?
**A**:
- SLURM / K8s 自动重启
- elastic training(PyTorch elastic)允许部分节点掉线
- 每 ~ 1k 步 checkpoint 一次
- 设置 NCCL 健康检查与超时

### Q5: 为什么 DeepSeek-V3 不用 TP?
**A**: 它有 MLA(§24)把 KV cache 压到 ~ 1/10,激活显存大幅下降;同时 MoE 的稀疏激活让单卡能装下完整 dense 部分。所以 TP=1,把通信预算留给 PP + EP。

## 14.14 本章小结

### 核心维度

```text
DP — 拆数据,All-Reduce 梯度
ZeRO — 切 optimizer / grad / param,跨卡分摊副本
TP — 切单层矩阵,All-Reduce 激活(单机内)
PP — 切跨层 stage,P2P 传激活(跨节点)
SP — 切序列长度,降激活显存
EP — 切 MoE 专家,All-to-All 路由
DualPipe — 双向流水线 + 通信计算重叠
```

### 核心要点

| 要点 | 关键 |
| --- | --- |
| **DP** | 任意规模、最简单 |
| **ZeRO** | 大模型必备,FSDP 是 PyTorch 原生 |
| **TP** | 单机内 NVLink 才用 |
| **PP** | 跨节点用,1F1B / DualPipe 减气泡 |
| **EP** | MoE 必用,通信主导 |
| **组合** | 大规模训练是 3-5D 笛卡尔积 |

### 一句话总结

> **LLM 训练的扩展性来自多维并行的笛卡尔积:DP 切数据、TP 切单层矩阵、PP 切跨层 stage、EP 切 MoE 专家、SP 切序列长度、ZeRO/FSDP 切优化器副本;DeepSeek-V3 用 DualPipe 把双向 forward/backward 与 All-to-All 通信重叠,把 1F1B 的气泡几乎压平,是开源生态目前的最佳实践。**
