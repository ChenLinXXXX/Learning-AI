## 摘要

LLM 的能力 ≈ 一个模型架构 × 一份数据。架构(Part II)已被卷成事实标准,**数据成为决定模型质量的最大变量**。本章按"工业流水线"梳理预训练数据准备:

- 数据来源与版权(Common Crawl、代码、书、论文、合成)
- 清洗流水线:抽取 → 语言识别 → 质量过滤 → 去重 → tokenize → shuffle
- 关键工具:fastText 分类器、MinHash + LSH 去重、PII 检测、领域配比
- 评估方式:多样性指标、held-out PPL、消融训练
- LLaMA / GPT / DeepSeek 等公布的数据规模与配比

读完你能在 PB 级 Common Crawl 上勾勒一条可执行的数据管道。

## 11.1 为什么数据比模型更重要?

### 11.1.1 公开证据

| 现象 | 结论 |
| --- | --- |
| LLaMA-1 在同架构下打败更大的 GPT-3 | 训练 token 数 ×6(1T vs 300B),架构次要 |
| Phi 系列(微软)用"教科书级数据"小模型打大 | 数据质量 > 数据规模 |
| 同模型在"加 vs 不加代码"上对推理能力差异巨大 | 数据混合配比是隐形旋钮 |
| LLaMA-3 用 15T token,远超 Chinchilla 建议的 ~140B | 大模型也"喂多了变好"(数据是廉价护城河) |

### 11.1.2 数据的"三种质量"

| 维度 | 含义 | 控制手段 |
| --- | --- | --- |
| **规模** | 总 token 数 | 多来源抓取 |
| **质量** | 单条样本的语义价值 | 过滤、合成、人工标注 |
| **多样性** | 覆盖语言、风格、领域、视角 | 配比、去重、采样 |

后续每节都对应这三件之一。

## 11.2 数据来源:CC / 代码 / 书 / 论文 / 中文 / 合成

### 11.2.1 Common Crawl(CC)

每月抓取的网页 dump,**~petabyte 规模、年覆盖**。开源派系 LLM 90% 文本来自 CC 的清洗版本(C4、RefinedWeb、Dolma、FineWeb…)。

### 11.2.2 代码

- GitHub 抓取(去 fork、按 license 过滤)
- The Stack v2(3.3T token,1000+ 语言)
- 代码数据对**推理 / 数学 / Agent** 能力收益巨大,主流 LLM 训练加 20-30% 代码

### 11.2.3 书籍 / 论文

- Project Gutenberg、Books3(版权争议大)
- arXiv、PubMed、专利
- 高质量长文本,对长上下文与逻辑链有显著正向作用

### 11.2.4 中文 / 多语言

- CC 中文占比 ~3-5%
- 国内来源:百度百科、知乎、微信公众号、政府公开文本
- 国际:Wikipedia 320+ 语言版本、CC 多语言、OPUS 平行语料

### 11.2.5 合成数据

近年的热点:**用强模型生成数据训弱模型 / 自蒸馏**。

- Microsoft Phi 系列:用 GPT-4 合成教科书级数据
- DeepSeek:V3 用 R1 蒸馏 reasoning 数据
- 风险:可能引入幻觉偏差、放大某些偏见,且涉及商业模型条款限制

### 11.2.6 数据规模量级

| 模型 | 训练 token 数 |
| --- | --- |
| GPT-2 | 40 GB(~10B token) |
| GPT-3 | 300 B |
| LLaMA-2 7B/13B/70B | 2 T |
| LLaMA-3 系列 | **15 T** |
| DeepSeek-V3 | 14.8 T |

## 11.3 流水线总览:Crawl → Extract → Filter → Dedup → Tokenize → Shuffle

```text
[Common Crawl WARC]
    │
    ▼
[正文抽取 trafilatura / boilerpipe]
    │
    ▼
[语言识别 fastText 176 lang]
    │
    ▼
[启发式质量过滤(长度、标点、字符比、关键词)]
    │
    ▼
[分类器质量过滤(高质量 vs CC 二分类)]
    │
    ▼
[精确去重(SHA256 行哈希)]
    │
    ▼
[近重复去重(MinHash + LSH)]
    │
    ▼
[毒性 / PII / 安全过滤]
    │
    ▼
[按领域 / 语种配比采样]
    │
    ▼
[Tokenizer 编码 → 二进制 shard]
    │
    ▼
[全局随机 shuffle]
    │
    ▼
[训练 dataloader]
```

每一阶段都可独立并行(MapReduce / Spark / Ray),工程上是真正的"大数据"问题。

## 11.4 语言识别与文本抽取

### 11.4.1 文本抽取

CC 的 WARC 文件是原始 HTML。要的不是导航栏、广告、JS,而是**正文**。常用:

- `trafilatura` / `readability-lxml` / `boilerpipe`:抽正文 + 标题 + 时间
- 移除 HTML 实体、规范化 Unicode(NFC / NFKC)

### 11.4.2 语言识别(LID)

`fastText` 的 `lid.176.bin` 模型:输入字符串,输出 176 种语言中最可能的一种 + 置信度。

工程要点:

- 短文本(< 100 字符)置信度低,直接丢
- 多语言混合文本(中英混排)按主导语种归类
- 维基同名词项跨语种需小心(可能误判)

### 11.4.3 输出形式

每条样本变成 JSON Lines:

```json
{"text": "...正文...", "url": "...", "lang": "zh", "score": 0.98}
```

进入下一阶段。

## 11.5 质量过滤:启发式 + 分类器 + 困惑度

### 11.5.1 启发式规则(C4 / Gopher 风格)

| 规则 | 阈值示例 |
| --- | --- |
| 字数 / token 数下限 | < 50 词 → 丢 |
| 单字符重复比例 | > 0.3 → 丢 |
| 标点符号比例 | < 0.005 或 > 0.5 → 丢 |
| 含 "Lorem ipsum" / 模板词 | 丢 |
| 列表行比例 | > 0.9(可能是导航/SEO 文本)→ 丢 |
| 含黑名单域名 / 关键词 | 按业务定义 |

经验:启发式过滤虽简单,但能砍掉 30-50% 垃圾,显著优于不过滤。

### 11.5.2 分类器过滤(GPT-3 / LLaMA 风格)

把"高质量参考语料(Wiki + Books)"作正类,把 CC 作负类,训 fastText 二分类器,推理时给每条 CC 文本打分,只保留 top-X%。

这种"用模型判定模型的食物"模式在 LLaMA-1 / Falcon / DeepSeek 都用过。

### 11.5.3 困惑度过滤

用一个小语言模型对每条文本算 PPL:

- 太高:可能乱码 / 罕见字符
- 太低:可能模板 / 重复

留中间分位段(如 25%-95%)。开销大,常作为最后精修。

### 11.5.4 主流配方

| 模型 | 主过滤手段 |
| --- | --- |
| GPT-3 | 启发式 + fastText 分类器 |
| LLaMA-1 | 启发式 + CCNet pipeline + fastText 分类器 |
| RefinedWeb / Falcon | 重启发式 + 极强去重,无分类器 |
| FineWeb-Edu(2024) | 用 Llama-3 给教育性打分,保留 top |

## 11.6 去重:exact / near / 语义

### 11.6.1 为什么去重最重要?

实证:**重复文本 1 次出现 ≈ 重复 N 次**——重复样本被模型反复看到,放大其偏差并降低 generalization。Lee et al.(2022)发现去重让模型困惑度提升 ~10%。

### 11.6.2 精确去重(exact dedup)

- 把每条文本(或长段落)SHA-256 哈希
- 用 hashset 或 Bloom filter 判重
- 工程上可在单台机器内放 ~ TB 级哈希(用 RocksDB / LevelDB)

抓不到"几乎相同但有标点差异"的重复 → 引入下一步。

### 11.6.3 近重复去重(MinHash + LSH)

**MinHash**:每条文档 → $k$ 个独立 hash 函数 → 取 shingles(n-gram)在每个 hash 下的最小值 → $k$-维签名向量。两个文档的 Jaccard 相似度可由签名重合率无偏估计。

**LSH**:把 $k$ 维签名分成 $b$ 段,每段哈希到桶——只比较落在同桶的文档对,$O(N)$ 找出所有候选近邻。

典型参数:$k = 128$,shingles = 5-gram。LLaMA、RefinedWeb、Dolma 都用这套。

### 11.6.4 实战发现

- **跨文档去重** > 文档内去重
- **段落级去重** 比文档级更激进,效果更好
- 顺手去掉 boilerplate(版权声明、Cookie 政策等),这类内容会污染整个 CC

### 11.6.5 语义去重(可选)

把文本 embedding 后聚类,合并近义但表述差异大的样本。开销大,目前主要用于 SFT 数据(§15)而非预训练。

## 11.7 毒性 / 隐私 / 安全过滤

### 11.7.1 毒性

- Perspective API / 自训分类器,过滤 toxicity 极高的样本
- 注意:**过滤太严会导致模型在某些场景"装聋作哑"** → 通常仅过滤极端,温和的偏见保留(由 RLHF / DPO 在后期处理,§16)

### 11.7.2 隐私 / PII

- 手机号、身份证、信用卡号、邮箱:用正则 / NER 检测,**遮盖**(替换为占位符)而非整条删除
- HuggingFace 的 `presidio`、`pii-checker` 等开源工具

### 11.7.3 版权与许可

- 训练数据合法性是 2023+ 的最大行业风险
- 工程做法:记录每条样本来源、license,出问题时可"按 source 回滚"
- 部分团队完全用 CC0 / 公共域 + 自有数据(EleutherAI、AllenAI)

| 领域 | 典型占比(LLaMA-1) |
| --- | --- |
| CommonCrawl + C4 网页 | 67% |

| Wikipedia 多语言 | 4.5% |

| StackExchange | 2% |
| 其他 | 余下 |

### 11.8.2 课程学习(Curriculum)

近年趋势:**训练后期切换到更高质量子集**,如 LLaMA-3 在最后 40B token 用人工筛选过的高质数据"退火"。

### 11.8.3 多语言权重

非英语语言常按 $p^\alpha$($\alpha \approx 0.5$)做温度采样,**平衡"主语种数据足够 vs 弱语种不被淹没"**。

## 11.9 数据评估:多样性 / 困惑度 / Held-out

### 11.9.1 多样性指标

- **去重前后 token 数比**:压缩比可直观看"原始 vs 唯一"信息量
- **领域分布**:用领域分类器统计实际配比是否符合预期
- **n-gram 覆盖率**:n-gram 唯一数 / 总 n-gram 数,衡量内容丰富度

### 11.9.2 困惑度评估

留一个 held-out 验证集(同分布):

- 用小模型在 held-out 上算 PPL,作为数据"易学度"度量
- 跨配方对比(加 vs 不加 X 来源)选更低 PPL 配方

### 11.9.3 数据规模与磁盘账

经验:**清洗后 1T token ≈ 4-6 TB 原始文本** ≈ 0.5-1 TB tokenized 二进制(uint16 时 2 bytes/token)。

LLaMA-2 训练 2T token → ~4 TB tokenized。Loader 设计要支持大 shard 文件 + 全局 shuffle(详见 §13)。

### 11.9.4 数据消融

最严格的"是否值得加这批数据":用同一模型同步训两次,只差此批数据,比较最终 benchmark。代价昂贵,通常只对关键决策做。

## 11.10 完整代码实现:一个 mini 数据清洗管道

```python
"""
mini_pipeline.py — 教学版预训练数据管道
依赖: pip install fasttext datasketch trafilatura
"""

import re
import hashlib
from datasketch import MinHash, MinHashLSH

# 1. 文本抽取(略,真实用 trafilatura.extract)

# 2. 启发式质量过滤
WORD_RE = re.compile(r"\w+")

def heuristic_keep(text: str) -> bool:
    words = WORD_RE.findall(text)
    if len(words) < 50: return False
    # 单字符重复
    most = max(text.count(c) for c in set(text))
    if most / max(len(text), 1) > 0.3: return False
    # 标点占比
    punct = sum(1 for c in text if not c.isalnum() and not c.isspace())
    if punct / max(len(text), 1) > 0.4: return False
    return True


# 3. 精确去重
seen_sha = set()

def exact_dedup(text: str) -> bool:
    h = hashlib.sha256(text.encode("utf-8")).hexdigest()
    if h in seen_sha: return False
    seen_sha.add(h)
    return True


# 4. 近重复去重(MinHash + LSH)
lsh = MinHashLSH(threshold=0.8, num_perm=128)
mh_id = 0

def shingles(text: str, k: int = 5):
    words = WORD_RE.findall(text.lower())
    return {" ".join(words[i:i+k]) for i in range(len(words) - k + 1)}

def near_dedup(text: str) -> bool:
    global mh_id
    m = MinHash(num_perm=128)
    for sh in shingles(text):
        m.update(sh.encode("utf-8"))
    if lsh.query(m): return False
    lsh.insert(f"d{mh_id}", m); mh_id += 1
    return True


# 5. 管道
def process(raw_docs):
    for doc in raw_docs:
        if not heuristic_keep(doc):    continue
        if not exact_dedup(doc):       continue
        if not near_dedup(doc):        continue
        yield doc


# === 演示 ===
docs = [
    "The quick brown fox jumps over the lazy dog. " * 20,        # 长度足够
    "The quick brown fox jumps over the lazy dog. " * 20,        # exact dup
    "The quick brown fox jumps over the LAZY dog! " * 20,        # near dup
    "aaaaa" * 200,                                                # 单字符重复 → 丢
]
kept = list(process(docs))
print(f"in={len(docs)} kept={len(kept)}")
```

每段都可单独运行验证。生产规模需把哈希集合换 RocksDB,把循环换成 MapReduce / Spark。

## 🎨 11.10.A 通俗类比合集

### 类比一: 烹饪准备 🍲

> 训练 LLM 像炖一锅好汤:模型架构是锅,数据是食材。

- **Crawl** = 去市场扫货,什么便宜什么贵都先买回来
- **Extract** = 去皮去骨,只留下能吃的部分
- **Quality Filter** = 挑掉烂菜叶、变质肉
- **Dedup** = 同一摊位买了 5 把一样的葱,只留 1 把(否则汤就一股葱味)
- **配比** = 主菜 / 配菜 / 调料的比例;代码就像辣椒,加一点提鲜,加太多齁
- **Tokenize** = 切块下锅,该切丁切丁该切片切片
- **合成数据** = 自制高汤替代部分原材料,鲜但要小心成本

### 类比二: 图书馆采购 📚

> 给图书馆采购藏书,目标是让读者获得最丰富、最准确的知识。

- **来源选择**:不能只买地摊小说,得有专著、期刊、教材并存
- **去重**:同一本《三国演义》买 100 本浪费,留几本就好
- **质量过滤**:盗版、印刷错版、错别字过多 → 退货
- **配比**:文学:科技:历史:哲学 按读者比例配
- **隐私**:涉及他人隐私的日记、未公开手稿 → 不收
- **退役**:旧版被新版取代时,把旧版下架

### 类比三: 拌沙拉 🥗

> 训练数据 = 一大盆混合沙拉。

- **生菜底(CC 网页)**:量大但味道平,得用其他食材提味
- **鸡胸肉(代码 + 数学)**:蛋白质,直接决定营养密度
- **坚果(书 + 论文)**:贵但精,加少量提质
- **酱料(配比)**:多大比例混合决定整体风味
- **手感(去重)**:不能整盘都是同一种,顾客一吃就腻
- **合成数据** = 自制 dressing,可控,但要小心不要靠 dressing 掩盖食材本身缺陷

### 三个类比对比

| 类比 | 强调什么 | 适合记忆什么 |
| --- | --- | --- |
| 🍲 烹饪 | **流程顺序** | 完整管道每一步 |
| 📚 图书馆 | **来源 + 退役** | 长期数据治理 |
| 🥗 沙拉 | **比例与风味** | 领域配比 / 合成数据 |

**记忆口诀**:
> **"抓 → 抽 → 过 → 去 → 切 → 配 → 洗(shuffle):一句话七步走。"**

## 11.11 常见疑问

### Q1: 我自己微调 7B 模型,需要这么复杂的数据准备吗?
**A**: 不需要完整流水线。微调常用 SFT 数据(§15),量级 1k-1M 条,人工或少量启发式过滤即可。本章的工业管道是为预训练 100B+ token 设计的。

### Q2: 去重是越激进越好吗?
**A**: 不是。极激进去重(Jaccard 阈值 0.5)会误伤真正的"多样化复述"(如不同新闻媒体报道同一事件),损失合法多样性。0.8-0.85 是经验甜点。

### Q3: 合成数据训出的模型会"塌缩"吗?
**A**: 有"model collapse" 风险——纯用模型生成数据迭代训练 N 代后,模型分布会逐步窄化。实证:**只要混合至少一定比例的人类真实数据,塌缩可缓解**。Phi 系列大量用合成数据 + 高质量人类种子,显示混合策略 viable。

### Q4: 如何评估"数据是否够用"?
**A**: 看 Chinchilla 比例 $D / P$:< 10 表示训练不足,~ 20 接近 Chinchilla 最优,> 100 进入"过训练"(LLaMA-3 路线,性价比角度选择)。详见 §17。

### Q5: 中文数据怎么处理?
**A**: 
- CC 中文有大量"农场内容"(关键词堆砌、机翻),启发式过滤要更激进
- 简繁混合:统一转简体或保留原貌,二选一
- 中文分词不是必须(BBPE 直接处理 UTF-8),但中文专属 tokenizer(§05.9)能省 token

## 11.12 本章小结

### 核心流程

```text
Crawl → Extract → LangID → Heuristic → Classifier
     → Exact Dedup → Near Dedup → PII / Toxicity
     → Domain Mix → Tokenize → Shuffle → Train
```

### 核心要点

| 要点 | 关键 |
| --- | --- |
| **数据 > 架构** | 同架构下,数据决定上限 |
| **去重最关键** | 比过滤更影响模型质量 |
| **MinHash + LSH** | 近重复检测标配 |
| **领域配比** | 代码 / 数学是推理能力开关 |
| **合成数据** | 加速 / 提质,但需混真实数据防塌缩 |

### 一句话总结

> **预训练数据是 LLM 真正的护城河;一条标准管道是"抓取 → 抽取 → 启发式过滤 → 分类器过滤 → 精确去重 → MinHash 近重复去重 → PII/毒性过滤 → 领域配比 → tokenize → shuffle";去重比过滤更重要,代码/数学/合成数据是现代配方中性价比最高的"调味料"。**

## 11.14 实战练习

### 练习 1 (★)

下载一份 CC News 的小子集(几 GB),用 `trafilatura` 抽正文,统计抽出文本的平均长度、语言分布。

### 练习 2 (★★)

### 练习 3 (★★)

写一个 MinHash + LSH 去重脚本,在 1GB CC 子集上对比 num_perm = 64 / 128 / 256 时:运行时间、内存占用、被识别为重复的比例。

### 练习 4 (★★★)

设计实验:在同一小模型(125M)上,固定其他变量,只改变"是否做近重复去重",训练 10B token,比较 held-out PPL 与 benchmark 分数。

### 练习 5 (★★★)

读 RefinedWeb / Dolma 的数据卡(data card),复现其过滤规则的至少 5 条,讨论你想保留/取消其中哪些。
