## 13.1 一次训练 step 的 6 个动作

```text
1. 取一个 batch          (dataloader.next)
2. 前向 forward         (model(x))
3. 计算 loss             (cross-entropy)
4. 反向 backward         (loss.backward())
5. 梯度裁剪 + optimizer  (clip + optimizer.step)
6. 调度 + 清梯度 + log   (scheduler.step / zero_grad / log)
```

错一步就崩。新手最常踩的坑:**第 6 步 zero_grad 忘了** → 梯度累积变 batch 翻倍且不可知。

## 13.2 训练循环骨架

```python
for step in range(total_steps):
    batch = next(data_iter)                       # 1. 取数据
    with torch.amp.autocast("cuda", dtype=torch.bfloat16):
        logits = model(batch.input_ids)           # 2. forward (mixed precision)
        loss = compute_loss(logits, batch.labels) # 3. loss
    loss.backward()                               # 4. backward
    grad_norm = torch.nn.utils.clip_grad_norm_(
        model.parameters(), max_norm=1.0)         # 5a. clip
    optimizer.step()                              # 5b. update
    scheduler.step()                              # 6a. lr schedule
    optimizer.zero_grad(set_to_none=True)         # 6b. zero
    if step % log_every == 0:
        log({"loss": loss.item(),
             "grad_norm": grad_norm.item(),
             "lr": scheduler.get_last_lr()[0]})   # 6c. log
```

整套主循环不超过 20 行——大部分复杂性在数据 / 分布式 / 容错的外围。

## 13.3 混合精度训练:bf16 / fp16 / FP8

### 13.3.1 为什么要混合精度?

- 显存:bf16 比 fp32 省一半,FP8 再省一半
- 算力:Tensor Core 在低精度下吞吐量翻倍
- 但低精度精度损失,需要混合策略

### 13.3.2 fp16 + loss scaling(2017 派)

fp16 范围窄($\sim \pm 6.5 \times 10^4$),梯度容易 underflow:

```python
scaler = torch.amp.GradScaler()
with torch.amp.autocast("cuda", dtype=torch.float16):
    loss = compute_loss(...)
scaler.scale(loss).backward()       # loss × 128(动态)
scaler.unscale_(optimizer)
torch.nn.utils.clip_grad_norm_(...)
scaler.step(optimizer)
scaler.update()
```

复杂 + 偶尔 NaN 需要回滚。LLaMA 时代之后已被 bf16 替代。

### 13.3.3 bf16(现代默认)

bf16 与 fp32 共享 8 位指数 → 范围相同,只损失尾数精度。**不需要 loss scaling**:

```python
with torch.amp.autocast("cuda", dtype=torch.bfloat16):
    loss = compute_loss(...)
loss.backward()
optimizer.step()
```

LLaMA、DeepSeek、Mistral、Qwen 全部 bf16。

### 13.3.4 "Mixed" 的含义

bf16/fp16 主体 + 关键节点 fp32:

| 节点 | 精度 |
| --- | --- |
| Embedding 输入 | bf16 |
| Attention QKV / FFN | bf16 |
| RMSNorm 平方和 | **fp32**(避免精度爆炸,§09) |
| Softmax 内累加 | **fp32** |
| Loss 累计 / 平均 | **fp32** |
| Optimizer states(m, v, master θ) | **fp32** |

### 13.3.5 FP8 训练(DeepSeek-V3 等)

FP8 两种格式:E4M3(精度优先,范围窄)、E5M2(范围优先,精度差)。前向用 E4M3,反向用 E5M2,关键节点仍 bf16/fp32:

| 节点 | 精度 |
| --- | --- |
| 大矩阵乘 forward(Linear / Attn QK^T) | **FP8 E4M3** |
| 大矩阵乘 backward(dW) | **FP8 E5M2** |
| Activation / Residual 主干 | bf16 |
| Norm / Softmax / Loss | fp32 |
| Optimizer | fp32 |

收益:DeepSeek-V3 报告 **训练吞吐 × 1.6,显存 × 0.6**;前提是对 FP8 量化与去量化的工程细节(per-tile 缩放、scale 因子更新)做到极致。

### 13.3.6 BF16 vs FP8 是否替换 fp32 master?

**不替换**。Adam 的 m、v、master $\theta$ 始终保留在 fp32 中——否则更新积累会被精度毁掉。这是"训练显存 ≈ 12P bytes"的关键(§04.5.5)。

## 13.4 Gradient Accumulation 与有效 batch

### 13.4.1 动机

GPT-3、LLaMA 等用 ~ 4 M token / step 的 macro-batch。单卡显存装不下,需要把 batch 拆成 N 个 micro-batch,分次 forward+backward,**梯度累加**后再 step。

### 13.4.2 公式

$$
\text{有效 batch} = \text{micro\_batch\_size} \times \text{accum\_steps} \times \text{n\_GPU}
$$

### 13.4.3 代码骨架

```python
optimizer.zero_grad(set_to_none=True)
for micro in range(accum_steps):
    batch = next(data_iter)
    with torch.amp.autocast("cuda", dtype=torch.bfloat16):
        loss = compute_loss(model(batch.input_ids), batch.labels)
        loss = loss / accum_steps                 # 平均到等同 macro-batch
    loss.backward()                               # 梯度累加
# 这里才 clip + step
torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
optimizer.step()
scheduler.step()
```

### 13.4.4 注意

- **scheduler.step() 按 macro-batch 步进**,不要每个 micro 都调
- 用 DDP 时配合 `model.no_sync()` 把前 N-1 个 micro 的 all-reduce 省掉(只在最后一个 micro 同步)

## 13.6 Checkpoint:保存 / 恢复 / 平均

### 13.6.1 要保存什么

```python
torch.save({
    "step": step,
    "model": model.state_dict(),         # fp32 master 或 bf16 sharded
    "optimizer": optimizer.state_dict(), # 含 m, v(fp32)
    "scheduler": scheduler.state_dict(),
    "rng_state": torch.get_rng_state(),
    "data_iter_state": data_iter.state_dict(),  # 自实现
}, f"ckpt-{step}.pt")
```

**漏一项就无法严格复现**。data_iter 状态(已消费多少 token、shuffle 种子)是新手最常漏的。

### 13.6.2 频率与保留策略

- 每 1000-5000 步保存一次
- 保留最新 N(如 3)份,自动清理旧的
- 关键里程碑(epoch 末、wallclock 整点)永久保留

### 13.6.3 权重平均(SWA / Model Soup)

后期 K 个 checkpoint 平均权重,常有 0.1-0.5 BPB 增益:

$$
\bar\theta = \frac{1}{K} \sum_{i=1}^{K} \theta_i
$$

LLaMA-3 在 decay 阶段做了 short-horizon averaging。

### 13.6.4 EMA(指数滑动平均)

训练过程中维护:

$$
\theta_{\text{ema}} \leftarrow \alpha \theta_{\text{ema}} + (1-\alpha) \theta
$$

$\alpha = 0.999$ 典型。LLM 训练大多不开(显存代价 +2P bytes,且收益小)。视觉模型常开。

## 13.7 日志与监控:loss / grad_norm / lr / tokens_per_sec

### 13.7.1 必看指标

| 指标 | 健康 | 危险 |
| --- | --- | --- |
| **loss** | 平滑下降 | 突跳、震荡、平坦 |
| **grad_norm** | 0.1-1.0 | > 10 警惕,> 100 必崩 |
| **lr** | 与调度一致 | 漂移意味着 step 数错 |
| **tokens/sec** | 与机器规格一致 | 突降意味着卡 / IO 出问题 |
| **GPU util** | > 80% | < 50% 说明 data / 网络瓶颈 |

### 13.7.2 工具

- W&B / TensorBoard / Aim 写入(本地或云)
- 自动告警:loss 不降 > 1k 步 或 grad_norm spike → push 通知

### 13.7.3 业务指标

每 N 万步在 held-out 上跑 PPL、HumanEval、MMLU 子集——loss 与下游能力的相关性不是 1,要双盲监控。

## 13.8 故障容错:断点续训、NaN 救援、损坏数据

### 13.8.1 断点续训

```python
if resume:
    ckpt = torch.load(latest_ckpt)
    model.load_state_dict(ckpt["model"])
    optimizer.load_state_dict(ckpt["optimizer"])
    scheduler.load_state_dict(ckpt["scheduler"])
    torch.set_rng_state(ckpt["rng_state"])
    data_iter.load_state_dict(ckpt["data_iter_state"])
    start_step = ckpt["step"] + 1
```

完美续训应**与不中断的训练曲线 bit-精确一致**。

### 13.8.2 NaN 救援

```python
if not torch.isfinite(loss):
    # 跳过此 step,记录数据样本,告警
    optimizer.zero_grad(set_to_none=True)
    continue
```

DeepSeek-V3 报告:全程 14T token 训练 **0 次回滚** —— bf16 + 严格 fp32 norm + grad clip 的功劳。

### 13.8.3 损坏数据

数据 shard 损坏时,loader 应**跳过 + 报警**,而不是整个训练崩溃。生产 loader 通常用 try/except + 数据 source 标记。

### 13.8.4 机器宕机

大规模训练用 SLURM / K8s 调度,设置:

- 心跳检测,死节点自动重启
- 训练脚本接 SIGTERM,优雅保存最近 checkpoint
- 节点损坏时自动从余下节点重平衡

## 13.9 完整代码实现:可运行 train loop

```python
import math
import torch
import torch.nn as nn
import torch.nn.functional as F

# 假设已有 model, data_iter, total_steps, accum_steps

optimizer = torch.optim.AdamW(model.parameters(), lr=3e-4,
                              betas=(0.9, 0.95), weight_decay=0.1)

def lr_lambda(step):
    warmup = 2000; floor = 0.1; peak = 1.0
    if step < warmup: return peak * step / warmup
    p = (step - warmup) / (total_steps - warmup)
    return peak * (floor + (1 - floor) * 0.5 * (1 + math.cos(math.pi * p)))

scheduler = torch.optim.lr_scheduler.LambdaLR(optimizer, lr_lambda)

model.train()
for step in range(total_steps):
    optimizer.zero_grad(set_to_none=True)
    loss_acc = 0.0
    for _ in range(accum_steps):
        batch = next(data_iter)
        with torch.amp.autocast("cuda", dtype=torch.bfloat16):
            logits, loss = model(batch.input_ids, batch.labels)
            loss = loss / accum_steps
        loss.backward()
        loss_acc += loss.detach()

    if not torch.isfinite(loss_acc):
        print(f"[step {step}] NaN/Inf, skipping")
        continue

    grad_norm = torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
    optimizer.step()
    scheduler.step()

    if step % 50 == 0:
        print(f"step={step:>6} loss={loss_acc.item():.4f} "
              f"lr={scheduler.get_last_lr()[0]:.2e} grad={grad_norm:.2f}")

    if step % 5000 == 0 and step > 0:
        torch.save({"step": step,
                    "model": model.state_dict(),
                    "optimizer": optimizer.state_dict(),
                    "scheduler": scheduler.state_dict()},
                   f"ckpt-{step}.pt")
```

## 13.10 现代 LLM 训练机房一日

一个真实的 LLaMA-3 / DeepSeek-V3 量级训练:

- **集群**: 数千 H100,InfiniBand 全连
- **吞吐**: 数百万 token / sec(集群级)
- **每 step**: 5-30 秒,取决于 model 大小与 batch
- **每天**: 几千亿 token,前进总训练数据的 1-3%
- **可见问题**: 5 min 一次硬件随机故障(GPU 报错、网卡丢包)
- **监控墙**: 全屏 loss / throughput / NaN / heartbeat

**训练成本**(以 DeepSeek-V3 671B / 14.8T token 为例):公开报告约 **5.5M H800 GPU-小时,总成本 ~ $5.5M USD**(对照 GPT-4 公开估算 $100M+)。

## 🎨 13.10.A 通俗类比合集

### 类比一: 长跑马拉松 🏃‍♂️

> 训练 LLM 像跑一场几周不停的超长马拉松。

- **每一步 step** = 跑步的一脚
- **warmup** = 起跑前热身
- **cosine 衰减** = 后程逐渐放慢节奏防受伤
- **grad clip** = 不管多急,单脚迈幅不能超过 1 米(防扭伤)
- **gradient accumulation** = 喘不过气时减小单脚迈幅,但多迈几脚再休息
- **checkpoint** = 跑到补给站存档(万一晕倒能从这里继续)
- **NaN skip** = 偶尔踩到水坑(异常 batch),抬腿跨过去就好,不停跑

### 类比二: 工厂 24×7 流水线 🏭

> 一个昼夜不停的产品流水线。

- **forward** = 上料 → 各工位加工
- **backward** = 检验员沿流水线倒走,标记每个工位的瑕疵贡献
- **optimizer.step** = 各工位按瑕疵反馈微调参数
- **混合精度** = 重活用 8 寸钻头(FP8),关键焊点用 16 寸(bf16/fp32)
- **gradient checkpointing** = 不全程录像,只在关键节点拍照,出问题时回放重做
- **监控** = 中央控制室盯着所有传感器,异常自动停线

### 类比三: 银行 ATM 现金运送 💰

> 严密、容错、可审计。

- **micro-batch** = 一辆运钞车,容量有限
- **gradient accumulation** = 多辆运钞车合并入库一次
- **checkpoint** = 每小时打印一份金库账目
- **断点续训** = 系统重启后,账目与凭证完全一致
- **NaN 救援** = 发现一笔可疑账(异常梯度),撤回该笔但不影响整体
- **FP8 vs bf16** = 普通客户币种用便宜账本(FP8),金条记账用高规格(fp32)

### 三个类比对比

| 类比 | 强调什么 | 适合记忆什么 |
| --- | --- | --- |
| 🏃 马拉松 | **节奏与持久** | warmup + cosine + 持久训练 |
| 🏭 流水线 | **forward / backward** | 训练 6 步动作顺序 |
| 💰 ATM | **容错与审计** | checkpoint / 容错 / 精度选择 |

**记忆口诀**:
> **"取-前-损-反-裁-更",六字步骤一气呵成;bf16 主干,fp32 兜底,FP8 加速。"**

## 13.11 常见疑问

### Q1: 为什么 fp16 时代用 loss scaling,bf16 不用?
**A**: fp16 指数 5 位,最小可表数 ~$6 \times 10^{-5}$,梯度容易 underflow → 必须放大。bf16 指数 8 位与 fp32 同,最小数 ~$10^{-38}$,几乎无 underflow,直接训即可。代价是尾数 7 位精度低。

### Q2: 训练时 model.train() 与 model.eval() 有什么差别?
**A**: 影响 Dropout、BatchNorm 等"训练/推理行为不同"的层。LLM 几乎不用 Dropout(§04.7.2)且无 BN,所以两者几乎无差别。但仍建议显式调用,养成习惯。

### Q3: 为什么 zero_grad 推荐 set_to_none=True?
**A**: 把 grad 直接置 None 比置 0 省一次写入显存的开销,且让 optimizer 跳过未参与 forward 的参数(MoE / 部分冻结时常见)。是 PyTorch 2.0+ 推荐做法。

### Q4: grad_norm 突然飙高怎么办?
**A**: 顺序排查:
1. 数据中有异常长 / 异常 token?
2. lr 是否过大(/2 试)?
3. RMS 是否未上 fp32(数值爆炸)?
4. 某层权重初始化是否过大?
单次 spike 配合 grad clip 通常无害,持续 spike 必须停训查根因。

### Q5: 训练中我要不要看下游 benchmark?
**A**: 是的,但**频率低**(每 5-10k 步一次)。loss 下降 ≠ 能力线性提升(常出现"能力涌现")。下游指标和 loss 互补监控,任一异常都该警觉。

## 13.12 本章小结

### 核心骨架

```text
取数据 → 前向(autocast bf16) → loss → backward
   → grad clip(1.0)→ optimizer.step → scheduler → zero_grad → log
```

### 核心要点

| 要点 | 关键 |
| --- | --- |
| **混合精度** | bf16 主干 + fp32 关键节点 + (可选 FP8) |
| **Grad accumulation** | 用算力凑 macro-batch |
| **Checkpoint** | 保存模型 + 优化器 + RNG + data iter 状态 |
| **监控** | loss / grad_norm / lr / tokens-per-sec |
| **容错** | NaN skip + 节点重启 + 优雅断点 |

### 一句话总结

> **训练循环只有"取-前-损-反-裁-更"六个动作,但每一步都要混合精度、梯度累积、checkpoint 与监控紧密配合。bf16 + AdamW + grad clip + warmup + cosine 已成为大模型训练的稳态配方;FP8 是 DeepSeek-V3 起的最新前沿,把成本再砍 40%。**

## 13.14 实战练习

### 练习 1 (★)

把 §13.9 的 train loop 接 §10.6 的 `GPTMini`,在 tiny shakespeare 跑 5000 步,记录 loss / grad_norm 曲线。

### 练习 2 (★)

实现 §13.6 的 checkpoint + resume:训练 2000 步保存 → 杀进程 → 从 ckpt 继续到 5000 步;对比"不中断"曲线,验证一致。

### 练习 3 (★★)

写一个"NaN 自动重试":遇到 NaN 时回滚到最近 ckpt,跳过 N 个 batch 后继续。

### 练习 4 (★★)

在 §13.9 中加上 gradient accumulation(accum_steps=4),验证"有效 batch ×4"的 loss 曲线与一次性大 batch 几乎重叠。

设计一份"训练监控 dashboard" 草图:列出 15-20 个必看指标 + 报警阈值,配合 SLURM 通知。
