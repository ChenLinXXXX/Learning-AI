## 22.2 硬件账:能跑多大模型?

### 22.2.1 显存粗算公式

$$
\text{显存} \approx \frac{P \cdot \text{bits/param}}{8} + \text{KV cache} + \text{推理开销} (\sim 1\text{ GB}) \tag{22.1}
$$

KV cache 见 §19。bits/param:bf16=16,INT8=8,Q4_K_M ≈ 4.5,INT4=4。

### 22.2.2 速查表

| 模型 | 精度 | 权重 | + KV(4k) | 推荐显存 |
| --- | --- | --- | --- | --- |
| 1.5B | bf16 | 3 GB | 0.1 GB | 6 GB |
| 7B | bf16 | 14 GB | 0.5 GB | 20 GB |
| 7B | Q4 | 4 GB | 0.5 GB | **6 GB(消费级!)** |
| 13B | Q4 | 8 GB | 0.8 GB | 12 GB |
| 32B | Q4 | 18 GB | 2 GB | 24 GB |
| 70B | Q4 | 35-40 GB | 4 GB | 48 GB |
| 70B | Q4 | — | 32k 上下文 | 80 GB(H100) |
| DeepSeek-V3 671B | FP8 | ~700 GB | (MLA 极小) | 8×80 GB |

### 22.2.3 常见显卡对照

| 卡 | 显存 | 带宽 | 算力(bf16) | 适合 |
| --- | --- | --- | --- | --- |
| RTX 3060 12GB | 12 | 0.36 TB/s | 12 TFLOPS | 7B Q4 |
| RTX 4090 24GB | 24 | 1.0 TB/s | 165 TFLOPS | 13B/32B Q4 |
| Mac M3 Max 64GB | 64(统一) | 0.4 TB/s | ~28 TFLOPS | 30B/70B Q4 |
| RTX 6000 Ada 48GB | 48 | 0.96 TB/s | 91 TFLOPS | 70B Q4 |
| A100 80GB | 80 | 2.0 TB/s | 312 TFLOPS | 70B bf16 / 多并发 |
| H100 80GB | 80 | 3.35 TB/s | 989 TFLOPS | 70B 商业部署 |
| H200 / B200 | 141-192 | 4.8+ TB/s | 1+ PFLOPS | 大模型生产 |

CPU(Mac M 系列除外)只能跑慢但能跑大;统一内存(Apple)是个人玩大模型的捷径。

## 22.3 量化格式总览:GGUF / AWQ / GPTQ / EXL2 / FP8

### 22.3.1 GGUF(llama.cpp)

- 全平台(CPU / GPU / Mac Metal / Vulkan)
- 子格式众多:Q2_K、Q3_K_S/M/L、Q4_K_S/M、Q5_K_M、Q6_K、Q8_0、F16…
- **Q4_K_M** 是性价比甜点(~ 4.5 bit/参数,质量损失 < 0.5 PPL)
- 优势:跨硬件、单文件、社区资源最丰富
- 缺点:GPU 上速度略不如 AWQ/GPTQ kernel

### 22.3.2 AWQ(Activation-aware)

- GPU(CUDA / ROCm)
- 4-bit,activation-aware,精度稳
- vLLM / TGI / TRT-LLM 直接支持
- 已成开源 GPU 部署主流

### 22.3.3 GPTQ

- 4-bit / 3-bit / 8-bit
- OBQ 系列误差最小
- 适合多种引擎
- 与 AWQ 平分秋色,选哪个看具体模型 release

### 22.3.4 EXL2(ExLlamaV2)

- 4-bit + 任意位宽混合(2.4 bit、3.0 bit…)
- 极致显存友好(可"挤出"上下文)
- 兼容性窄(仅 ExLlamaV2 / oobabooga)

### 22.3.5 FP8

- H100 / B100 原生硬件支持
- 接近 bf16 精度、2× 速度
- 商业部署主推

### 22.3.6 选型决策

| 场景 | 推荐 |
| --- | --- |
| 个人 / Mac / 笔记本 | **GGUF Q4_K_M / Q5_K_M** |
| 单 GPU 部署 | **AWQ-INT4** |
| 多 GPU 商业 | **AWQ / GPTQ / FP8** |
| H100+ | **FP8** |
| 极致显存压榨 | **EXL2 (2.5-3 bit)** |

## 22.4 推理引擎家族

### 22.4.1 llama.cpp

- C++,跨平台(Linux / Win / Mac / iOS / Android)
- CPU / Metal / CUDA / Vulkan / ROCm
- 主格式 GGUF
- 单二进制 `./llama-cli`,极简

### 22.4.2 Ollama

- llama.cpp 的"超友好封装",自动管理模型库 / OpenAI API server
- 一行命令:`ollama run qwen2.5:7b`
- 个人开发首选

### 22.4.3 LM Studio

- 桌面 GUI(Mac / Win / Linux),拖拽即装
- 适合不写代码的用户、做 demo

### 22.4.4 vLLM

- Python / CUDA / ROCm
- PagedAttention + Continuous Batching 事实标准
- OpenAI-Compatible Server 内置
- 部署单卡到千卡集群

### 22.4.5 SGLang

- 类似 vLLM 但重点:**结构化输出 + RadixAttention(prefix cache)**
- 高并发 chatbot / agent 场景吞吐更优
- 与 vLLM 互为补充

### 22.4.6 TGI(Text Generation Inference)

- HuggingFace 出品
- 与 HF 生态集成最丝滑
- vLLM 出现后市场份额下降,但仍稳定

### 22.4.7 TensorRT-LLM

- NVIDIA 官方
- 极致性能,商业首选
- 工程门槛高,需要 Triton Inference Server / 编译模型

### 22.4.8 MLX

- Apple Silicon 原生(M1-M4)
- 速度近原生 Metal
- 与 llama.cpp Metal backend 互补

### 22.4.9 DeepSeek 内部栈

- 不开源,但报告披露:**Disaggregated Serving + EP + DualPipe + 自研 kernel**
- 上限对标 TensorRT-LLM,代码风格更激进

## 22.5 实战配方一:个人 Mac / Windows / Linux

### 22.5.1 最简:Ollama

```bash
# 1. 安装(Mac/Win/Linux 都有官方安装包)
curl -fsSL https://ollama.com/install.sh | sh

# 2. 拉模型(自动选合适量化)
ollama pull qwen2.5:7b
ollama pull deepseek-r1:7b

# 3. 命令行对话
ollama run qwen2.5:7b

# 4. OpenAI 兼容 API(默认监听 11434)
curl http://localhost:11434/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model":"qwen2.5:7b","messages":[{"role":"user","content":"你好"}]}'
```

### 22.5.2 进阶:llama.cpp + GGUF

```bash
# 1. 编译(Mac 自动 Metal)
git clone https://github.com/ggerganov/llama.cpp && cd llama.cpp
make -j

# 2. 下 GGUF(HuggingFace 上找 *-GGUF 仓库)
huggingface-cli download Qwen/Qwen2.5-7B-Instruct-GGUF qwen2.5-7b-instruct-q4_k_m.gguf

# 3. 跑
./llama-cli -m qwen2.5-7b-instruct-q4_k_m.gguf -p "你好" -n 200

# 4. 起 OpenAI 兼容 server
./llama-server -m qwen2.5-7b-instruct-q4_k_m.gguf --port 8080 -c 8192 -ngl 99
```

参数要点:

- `-c`:上下文长度
- `-ngl`:offload 多少层到 GPU(Mac 上 99=全部)
- `-t`:CPU 线程数

### 22.5.3 MLX(Apple Silicon)

```bash
pip install mlx-lm
mlx_lm.server --model mlx-community/Qwen2.5-7B-Instruct-4bit --port 8080
```

M 系列 Mac 上 MLX 通常比 llama.cpp Metal 略快。

## 22.6 实战配方二:单卡 GPU(消费 / 企业)

目标:24-80 GB 单卡,7B-70B,中等并发(10-50)。

### 22.6.1 vLLM 一行起服务

```bash
pip install vllm

# AWQ-INT4 7B,24GB 单卡能跑 32k 上下文
vllm serve Qwen/Qwen2.5-7B-Instruct-AWQ \
  --max-model-len 32768 \
  --gpu-memory-utilization 0.9 \
  --quantization awq \
  --port 8000

# 70B AWQ 在 H100 80GB
vllm serve casperhansen/llama-3.3-70b-instruct-awq \
  --max-model-len 32768 \
  --gpu-memory-utilization 0.9 \
  --quantization awq
```

### 22.6.2 常用参数

| 参数 | 含义 | 建议 |
| --- | --- | --- |
| `--max-model-len` | 最大上下文 | 4k-128k |
| `--gpu-memory-utilization` | 显存利用率 | 0.85-0.95 |
| `--max-num-seqs` | 最大并发 | 32-256 |
| `--quantization` | 量化格式 | awq / gptq / fp8 |
| `--kv-cache-dtype` | KV cache 精度 | fp16 / fp8 / int8 |
| `--enable-prefix-caching` | 前缀缓存 | 多轮对话 / RAG 开 |
| `--enable-chunked-prefill` | chunked prefill | 长 prompt 开 |
| `--speculative-config` | speculative decoding | 大模型可配 |

### 22.6.3 SGLang(prefix cache 更强)

```bash
pip install "sglang[all]"

python -m sglang.launch_server \
  --model-path Qwen/Qwen2.5-7B-Instruct \
  --port 30000 \
  --enable-radix-cache
```

API 与 vLLM 类似,/v1/chat/completions 兼容 OpenAI。

## 22.7 实战配方三:多卡 / 多节点服务

### 22.7.1 Tensor Parallel(单机多卡)

```bash
# 4×A100 上 70B bf16
vllm serve meta-llama/Meta-Llama-3.1-70B-Instruct \
  --tensor-parallel-size 4 \
  --max-model-len 32768
```

注意:`tp` 数需整除 head 数;Inter-GPU 走 NVLink 才有意义。

### 22.7.2 Pipeline Parallel(跨节点)

```bash
vllm serve <large_model> \
  --pipeline-parallel-size 4 \
  --tensor-parallel-size 4
```

总 GPU = pp × tp。跨节点用 Ray 或 SLURM 启动。

### 22.7.3 DeepSeek-V3 671B 等大 MoE

需要专门的 EP 配置 + 多节点 + 高速互联(InfiniBand)。当前路径:

- vLLM 官方已支持 DeepSeek-V3(2024 末)
- 也可用 SGLang
- DeepSeek 官方开源了部分推理 kernel

### 22.7.4 Disaggregated Serving

vLLM 在 2024 加入实验性 prefill/decode 解耦;Mooncake / DistServe 等独立项目更激进。**只在大流量 / 严格 SLA 场景值得**。

## 22.8 OpenAI-Compatible API 与客户端集成

### 22.8.1 标准 endpoint

几乎所有引擎都支持:

- `/v1/chat/completions`
- `/v1/completions`
- `/v1/embeddings`(部分)
- `/v1/models`

### 22.8.2 Python 客户端

```python
from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:8000/v1",
    api_key="not-used",
)
resp = client.chat.completions.create(
    model="qwen2.5-7b",
    messages=[{"role":"user","content":"你好"}],
    temperature=0.7, max_tokens=200,
)
print(resp.choices[0].message.content)
```

### 22.8.3 主流 IDE / 工具

- Claude Code / Cursor / Cline:都支持自定义 base_url
- Open WebUI / LobeChat / NextChat:Web UI 接本地 endpoint
- LangChain / LlamaIndex:`OpenAI(base_url=...)` 即可

→ 本地模型几乎可无缝替换 OpenAI / Anthropic 在任何工具里。

### 22.8.4 流式输出

`stream=True` 即可拿 SSE 流。前端用 EventSource 接收,逐 token 渲染。

## 22.9 量化模型怎么自己出?

### 22.9.1 AWQ

```python
from awq import AutoAWQForCausalLM
from transformers import AutoTokenizer

model_path = "Qwen/Qwen2.5-7B-Instruct"
quant_path = "qwen2.5-7b-awq"

model = AutoAWQForCausalLM.from_pretrained(model_path)
tok = AutoTokenizer.from_pretrained(model_path)

quant_config = {"zero_point": True, "q_group_size": 128, "w_bit": 4, "version": "GEMM"}
model.quantize(tok, quant_config=quant_config)
model.save_quantized(quant_path)
tok.save_pretrained(quant_path)
```

校准数据用几百条 pile / wiki / 自有数据即可,AWQ 对校准数据不敏感。

### 22.9.2 GPTQ

```bash
pip install auto-gptq

# 或用 vLLM 配套的 llmcompressor
python -m llmcompressor quantize \
  --model-path Qwen/Qwen2.5-7B-Instruct \
  --output-dir qwen-gptq \
  --method gptq --bits 4
```

### 22.9.3 GGUF

```bash
# llama.cpp 内 convert + quantize
python convert_hf_to_gguf.py /path/to/model
./llama-quantize model-f16.gguf model-q4_k_m.gguf Q4_K_M
```

社区一般直接用 `TheBloke` / `bartowski` / `mradermacher` 等仓库分发的 GGUF。

### 22.9.4 PR-aware 校准

如果你的模型主要服务某领域(法律 / 医疗 / 代码),用该领域语料校准 → 精度损失更小。

## 22.10 监控、压测、安全

### 22.10.1 监控

- `nvidia-smi --query-gpu=utilization.gpu,memory.used --format=csv -l 1`
- vLLM `/metrics`(Prometheus 格式),含 token throughput / latency / queue
- 业务侧:p50/p95/p99 TTFT、TPOT、错误率、并发

### 22.10.2 压测

```bash
# vLLM 自带压测
python -m vllm.entrypoints.openai.benchmark \
  --model qwen2.5-7b \
  --num-prompts 1000 \
  --request-rate 10
```

或用 `wrk`、`locust`、`k6` 仿真真实流量。关注尾延迟,而非均值。

### 22.10.3 安全

- **网络**:本地服务不要 0.0.0.0 暴露公网;加 nginx + token auth
- **prompt 注入**:输入做 schema 校验、关键字过滤(详见 §32)
- **输出过滤**:辱骂 / PII / 越狱模式监控
- **日志**:输入输出最少必要保存(隐私法规)
- **资源限速**:per-IP rate limit,防 token 滥用

## 22.11 排错速查

| 症状 | 可能原因 | 对策 |
| --- | --- | --- |
| OOM | 上下文太长 / KV 没量化 | 减 max_model_len / KV INT8 / 换更小模型 |
| TTFT 很大 | 长 prompt 阻塞 | 开 chunked prefill |
| 输出乱码 | tokenizer 不匹配 / chat template 错 | 重检查 model_card |
| 复读 | 采样参数不当 | 加 repetition_penalty、降 temperature |
| GPU util < 50% | batch 太小 / IO 慢 | 加并发、检查数据 pipeline |
| 偶尔 NaN | bf16 数值边界 | RMSNorm fp32、降 LR(训练) |
| 速度时快时慢 | 调度抖动 / chunk 过大 | 调小 chunk、限速 |
| Mac 性能差 | 没用 Metal / mlx | `-ngl 99` 或换 MLX |
| 70B AWQ 输出质量低 | 量化损失 / 校准差 | 换 fp8 / 重新校准 |
| 多卡卡死 | NCCL 通信失败 | 检查 NCCL_DEBUG, GPUDirect P2P |

## 🎨 22.11.A 通俗类比合集

### 类比一: 装机比赛 🖥️

> 部署 = 配一台机器跑你的"大模型游戏"。

- **硬件账** = 给定预算和需求选 CPU/GPU/内存
- **量化** = 把游戏材质从超高画质降到中等,流畅度大涨,肉眼差异小
- **vLLM / Ollama** = 像 Steam / Epic,把"运行游戏"封装成一键启动
- **OpenAI 兼容 API** = 标准手柄接口,任何客户端都能接
- **监控** = 任务管理器:CPU、GPU、风扇噪音、温度
- **排错** = 蓝屏 / 黑屏 / 帧率掉 → 顺序检查驱动、显存、散热

### 类比二: 餐车开业 🚚

> 你要把"大模型"开成路边餐车。

- **小餐车(Ollama / llama.cpp)**:个人出摊,菜单简单,装备便宜
- **中等门店(vLLM 单卡)**:固定店面,5-50 个并发客人
- **连锁餐厅(多节点 vLLM / TRT-LLM)**:跨城市部署,流量大
- **预制菜(量化模型)**:节省厨师时间,客人吃不出差别
- **后厨监控(Prometheus)**:每个炉灶温度、上菜时间、客流
- **食材保鲜(KV cache 管理)**:不能让备料烂掉,但也不能囤太多

### 类比三: 翻译公司 🌐

> 大模型是会说多语言的翻译,客户找你做翻译服务。

- **隐私选项** = 本地翻译(机密)/ 私有云 / 公有云
- **量化** = 用经验丰富但稍快稿的译员(便宜 4×,差异不大)
- **多卡 TP** = 一个项目 4 个译员协作,各负责段落
- **多节点 PP** = 不同语种走不同部门,接力完成
- **OpenAI API 标准** = 公司用统一报价单,客户随便换公司也不用改流程
- **监控** = 翻译完工时间 / 错字率 / 客户投诉率

### 三个类比对比

| 类比 | 强调什么 | 适合记忆什么 |
| --- | --- | --- |
| 🖥️ 装机 | **硬件 + 软件搭配** | 选型决策 |

## 22.12 常见疑问

### Q1: 我只有一张 RTX 3060 12 GB,能跑 7B 吗?
**A**: 能。GGUF Q4_K_M,~ 5 GB 模型 + 2-4 GB KV(8k 上下文),用 llama.cpp / Ollama 流畅。

### Q2: Mac Studio 用 M3 Ultra 192GB 跑 70B 怎么样?
**A**: 完全可以。MLX 或 llama.cpp Metal,Q4_K_M 70B 内存 40-50 GB,速度 5-15 tok/s,人感觉接近"中等流畅"。

### Q3: Ollama vs llama.cpp 直接用?
**A**:
- Ollama:零门槛、易管理、内置 API
- llama.cpp 直用:控制更细、性能略好、模型来源自由

### Q4: vLLM 启动总报"Out of Memory"?
**A**:
- 减 `--max-model-len`
- 加 `--gpu-memory-utilization 0.95`(若 GPU 独占)
- 用量化模型
- 启用 `--kv-cache-dtype fp8`(H100)或 `int8`
- 减 `--max-num-seqs`

### Q5: 我可以同时跑训练和推理在同一卡吗?
**A**: 不建议。两者同卡互相挤压,延迟与吞吐都崩。生产上分卡或分集群。

### Q6: 哪个开源模型现在最值得本地跑?
**A**(2026 视角):
- 性价比通用:Qwen-2.5 / 3 系列、Llama-3.3
- reasoning:DeepSeek-R1 / DeepSeek-V3.x
- 长上下文:Qwen-2.5-1M、Yi-1.5
- 代码:DeepSeek-Coder-V2 / Qwen-Coder

## 22.13 本章小结

### 核心选型矩阵

```text
个人 / Mac        → Ollama / llama.cpp / MLX + GGUF Q4_K_M
单 GPU 消费       → vLLM + AWQ-INT4(7B-32B)
单 GPU 企业 H100  → vLLM + FP8 / AWQ(70B)
多卡 NVLink      → vLLM + TP
多节点           → vLLM + TP + PP / SGLang
极致商业服务      → TensorRT-LLM
大流量 SLA       → Disaggregated Serving(DistServe / Mooncake)
```

### 核心要点

| 要点 | 关键 |
| --- | --- |
| **硬件账** | 公式 22.1 估算 |
| **量化** | GGUF(跨平台)、AWQ(GPU 主流)、FP8(H100+) |
| **引擎** | 个人 Ollama / 服务 vLLM / 极致 TRT-LLM |
| **OpenAI 兼容 API** | 一切互通的协议 |
| **监控 / 压测 / 安全** | 上线前 checklist |

### 一句话总结

> **本地部署是"在硬件 / 模型 / 上下文 / 并发 / 隐私"五维空间里挑配方:个人玩家选 Ollama+GGUF,企业服务选 vLLM+AWQ,大流量上 TRT-LLM 或解耦架构,极致 H100 用 FP8;所有路径都收敛到一个 OpenAI 兼容 API,让上层应用无缝替换商业模型。**
