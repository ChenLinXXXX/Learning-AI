## 摘要

推理的瓶颈不是单一的"算力"或"显存",而是**算力 / 带宽 / IO / 调度 / 数值精度**多维博弈。本章把 5 大优化族讲透:

- **Flash Attention**:通过 tiling + 重算把 attention 压进 GPU SRAM,IO 砍掉 5-10×
- **Chunked Prefill**:把长 prefill 切块,与 decode 交错,平衡 TTFT/TPOT
- **Speculative Decoding**:小模型先猜 / 大模型一次验,接受率 ~80% 时吞吐 2-3×
- **MTP / Medusa**:让模型自带"猜测能力",DeepSeek-V3 的 speculative 来源
- **Sliding Window / Streaming-LLM**:超长上下文的"近似 attention"
- **Compile**:`torch.compile` / TensorRT / Triton kernel fusion
- **量化推理**:INT8 / INT4 / FP8 权重 + KV(§19)+ 激活

读完你能解释 vLLM、SGLang、TensorRT-LLM 内部各项优化的来源,并按 SLA 自由组合。

## 21.1 Roofline:算力 vs 带宽

### 21.1.1 模型

GPU 同时受**算力(FLOPS)**与**带宽(GB/s)**约束。对一段 kernel:

$$
\text{运行时长} \geq \max\!\left(\frac{\text{FLOPs}}{\text{TFLOPS}}, \frac{\text{Bytes moved}}{\text{HBM\_BW}}\right) \tag{21.1}
$$

按操作的"**算力强度**(FLOPs / Byte)"分类:

| 类型 | 算力强度 | 受限于 | LLM 代表 |
| --- | --- | --- | --- |
| **Compute-bound** | 高 | 算力 | 大矩阵乘(FFN、prefill 的 attention) |
| **Memory-bound** | 低 | 带宽 | LayerNorm、Softmax、**decode 的 attention** |

### 21.1.2 关键直觉

- **算力受限**操作:量化、加 Tensor Core 收益大
- **带宽受限**操作:reducing memory IO 收益大 → **Flash Attention 的根本动机**

### 21.1.3 具体数字

H100(SXM):

- bf16 算力 ~ 989 TFLOPS
- HBM3 带宽 ~ 3 TB/s
- 临界算力强度 ~ 330 FLOPs / Byte

LayerNorm 算力强度 ~ 2 FLOPs / Byte → **远在带宽受限区**。Attention 矩阵乘 ~ 上千 FLOPs / Byte → 算力区。但 decode 时 batch=1、序列短 → attention 算力强度退化到 ~ 2 → **重新跌回带宽区**。

## 21.2 Flash Attention:IO-aware 注意力

### 21.2.1 朴素 attention 的 IO 浪费

```text
Step 1: 算 S = Q K^T          → 写到 HBM
Step 2: 算 P = softmax(S)     → 读 S, 写 P
Step 3: 算 O = P V            → 读 P, 写 O
```

中间矩阵 $S, P \in \mathbb{R}^{n \times n}$ 反复进出 HBM,**IO ~ $O(n^2)$**,在长序列下成为瓶颈。

### 21.2.2 Flash Attention 思路(Dao et al., 2022)

1. **Tile**:把 Q, K, V 切成小块,每次只算一块 attention
2. **Online softmax**:增量地维护 $m_i$(running max)与 $\ell_i$(running sum)来不变 softmax 性质
3. **Fused kernel**:整个 attention 用一个 CUDA kernel,中间结果只在 SRAM(片上内存)流转,**不写回 HBM**

数学等价,IO 从 $O(n^2)$ 砍到 $O(n)$,实测 attention 整体加速 2-5×。

### 21.2.3 Flash Attention 2 / 3

- v2:更好的并行(沿 Q 维而非 KV 维)
- v3(H100 专用):用 Tensor Memory Accelerator (TMA) + warpgroup,bf16 接近峰值算力

### 21.2.4 工程使用

- PyTorch 2.0+ 的 `torch.nn.functional.scaled_dot_product_attention` 默认调用 Flash 后端
- vLLM / SGLang / TGI 全用
- 不需要修改模型代码,只需 PyTorch + CUDA 版本对得上

### 21.2.5 反向也能 Flash

Flash Attention 同时实现了 **backward 的 fused kernel**,在训练中也能省 5-10× 显存。这是为什么大模型训练几乎必开。

## 21.3 Chunked Prefill 与 Prefill-Decode 交错

### 21.3.1 长 prefill 的问题

用户传入 16k token 的 prompt → prefill 一次跑完要几秒 → **同 batch 的 decode 请求被卡几秒**,TPOT 抖动剧烈。

### 21.3.2 Chunked Prefill

把长 prompt 切成 512 / 1024 token 的 chunk,**每 step 只 prefill 一个 chunk**,与 decode 请求交错:

```text
Step 1: prefill chunk_1 of req_A + decode req_B,C,D...
Step 2: prefill chunk_2 of req_A + decode req_B,C,D...
...
Step k: prefill 完成 + decode req_A
```

TTFT 略升(总 prefill 时间不变),但**全局 TPOT 平稳**。

### 21.3.3 SLA 视角

- p99 TPOT 关键 → chunk_size 调小(如 256),抖动小
- p50 TTFT 关键 → chunk_size 调大,prefill 完成早

vLLM 默认 chunk_size = 512,可调。

### 21.3.4 与 PagedAttention 配合

chunked prefill 必须让 KV cache 支持"渐进式 append"(同请求的 KV cache 跨多 step 增长)→ PagedAttention 天然支持。

## 21.4 Speculative Decoding

### 21.4.1 思路

decode 是 memory-bound 的——**算力大量空闲**。能否一次性并行 forward 多个候选 token,看哪些和大模型预期一致,**接受多个 token 后才走一次大模型 forward**?

### 21.4.2 经典算法(Leviathan et al., 2023)

需要 **draft model**(快的小模型)+ **target model**(大模型):

```text
1. draft model 用 KV cache 顺序生成 K 个候选 token y_1...y_K
2. target model 一次 forward(并行算 K+1 个位置的 logits)
3. 按拒绝采样规则,逐位置判断"接受 vs 拒绝":
   - 若 p_target(y_i) >= p_draft(y_i): 接受 y_i
   - 否则按概率比 p_target/p_draft 接受;若拒绝,从 (p_target - p_draft)+ 重采
4. 接受到第一个拒绝位置;target 的拒绝处采样作为修正
```

数学保证:**生成分布与 target 完全一致**(无质量损失)。

### 21.4.3 实际效果

- **接受率** $\alpha$ 经验 50%-90%(draft 越准越高)
- 单次"target forward"产出 $\sim 1 + \alpha + \alpha^2 + \ldots + \alpha^K$ 个 token
- 等效加速 ~ $1/(1-\alpha)$,实测 2-3×

### 21.4.4 加速公式(简化)

记 target 单 forward 耗时 $T_t$,draft 单 forward $T_d \ll T_t$,K 个 draft step + 1 target step,期望接受 $\bar n$ token:

$$
\text{Speedup} \approx \frac{\bar n}{1 + K \cdot T_d / T_t} \tag{21.2}
$$

draft 越小越快 → 分母小;接受率越高 → 分子大。常取 $K = 4\text{-}8$。

### 21.4.5 Draft model 选择

- 同模型小尺寸版(如 LLaMA-7B 当 LLaMA-70B 的 draft)
- 同模型自蒸馏小版
- "**自身**":Medusa(下节)

### 21.4.6 重要陷阱

- draft 与 target 必须**共享 tokenizer**
- 采样参数(temperature, top_p)在二者上必须严格匹配
- batch 模式实现复杂:不同 seq 接受到不同位置 → 需要 ragged batching

## 21.5 Multi-Token Prediction(MTP)用作 speculative

### 21.5.1 Medusa(Cai et al., 2024)

在原模型上**外挂多个 MTP head**,每个 head 预测未来第 k 个 token。用 MTP 输出作 draft,target 一次验。

- 只需训练 head(小),不动主模型
- 接受率 70-85%

### 21.5.2 DeepSeek-V3 的 MTP

§12.5 已经讲过:训练时就用 MTP 作为辅助 loss,推理时这些 head 天然作 draft generator。接受率 ~85%,吞吐 +80%。

### 21.5.3 EAGLE / EAGLE-2(2024)

用一个轻量 transformer 在原模型隐藏态上预测多个 token,接受率高,工程化好,主流推理引擎(vLLM、SGLang)都已支持。

### 21.5.4 BLT / EAGLE3(2025)

更先进的"自 speculative"路线,接受率压到 90%+。这是 2025 年推理优化最活跃的方向。

## 21.6 Sliding Window / Streaming-LLM

### 21.6.1 动机

超长上下文(128k+)的 KV cache 装不下 → 想"丢弃远处历史"但又不能崩。

### 21.6.2 Sliding Window Attention(Mistral)

每个 token 只看最近 $w$ 个 token($w = 4096$ 典型)。**层数叠加产生类似 RNN 的有效感受野** $\sim L \cdot w$。

- 训练时就用此结构(Mistral / Gemma 等)
- KV cache 只保留窗口内的 token,显存 $O(w)$ 而非 $O(n)$

### 21.6.3 Streaming-LLM(Xiao et al., 2023)

发现 attention sink 现象:**最开始 4 个 token 的 attention 占比异常高**。只保留 "前 4 个 sink + 最近 W 个" 也能让 LLaMA 处理无限长流。

适合 chatbot streaming 场景。

### 21.6.4 局限

- 严格说会丢失远处信息,**不能替代真正长上下文**
- 数学 / 法律 / 长文档分析慎用
- 适合对话流式(老历史不重要)

## 21.7 Compile / Graph Optimization

### 21.7.1 torch.compile

PyTorch 2.0 起,用 `torch.compile(model)` 自动:

- Tracing 计算图
- Kernel fusion(把多个小算子融成一个 CUDA kernel)
- 静态形状特化
- 选最优 backend(Inductor → Triton)

通常推理加速 1.3-2×,**无需改模型代码**。

### 21.7.2 TensorRT-LLM

NVIDIA 官方:在 Triton / cuBLAS / cuDNN 之上,提供 Transformer 专用图优化、INT4/INT8/FP8 kernel、in-flight batching。极致性能但工程门槛高。

### 21.7.3 自写 Triton kernel

vLLM、Flash Attention、Liger Kernel 等都大量用 Triton(类 Python 写 GPU kernel):

- 可读性 >> CUDA
- 性能接近手写 CUDA
- 跨 GPU 厂商(AMD / Intel 也在支持)

### 21.7.4 何时该写自己的 kernel

- 算子奇怪(MLA、自定义 sampler)
- 在 hot path 上而库不优
- 否则用现成

## 21.8 量化推理:Weight / KV / Activation

### 21.8.1 权重量化(§22 详)

把 fp16/bf16 权重压到 INT8 / INT4。常见方案:

- **GPTQ**:layer-wise post-training,误差最小化
- **AWQ**:Activation-aware,激活大的 channel 用更高精度
- **HQQ / SmoothQuant**:再细分
- **NF4 / FP4**(QLoRA 派):4-bit 浮点形式

收益:

| 量化 | 显存 | 速度 | 精度损失 |
| --- | --- | --- | --- |
| W8A16 | 0.5× | 1.3-1.7× | ~0 |
| W4A16 | 0.25× | 2-3× | < 0.5 PPL |
| **FP8(H100 起)** | 0.5× | **2×** | < 0.1 PPL |

### 21.8.2 KV 量化(§19.5)

INT8 / INT4 / FP8 of K, V cache。与权重量化正交叠加。

### 21.8.3 Activation 量化

最难。SmoothQuant、AWQ 把"难量化的激活"通过缩放转移到权重侧。INT8 activation 量化在 H100 / B100 上能进一步加速。

### 21.8.4 实战配方

| 场景 | 推荐 |
| --- | --- |
| 单卡跑 7B | bf16 |
| 单卡跑 14B | INT8 / AWQ-INT4 |
| 单卡跑 70B | AWQ-INT4 / GPTQ-INT4 |
| H100 SXM | FP8(原生支持) |
| 边缘 CPU / Mac | GGUF Q4_K_M / Q5_K_M |

## 21.9 完整代码:Speculative Decoding 最小版

```python
import torch
import torch.nn.functional as F

@torch.no_grad()
def speculative_decode(target, draft, input_ids, max_new=64, K=5,
                       temperature=1.0):
    """
    最小可读 speculative decoding,bs=1。
    target, draft: 实现 forward(input_ids) -> logits;支持 KV cache(略)
    """
    device = input_ids.device
    out = input_ids

    while out.size(1) - input_ids.size(1) < max_new:
        # 1. draft 顺序生成 K 个候选
        draft_tokens = []
        draft_probs  = []
        cur = out.clone()
        for _ in range(K):
            logits = draft(cur).logits[:, -1, :] / temperature
            p = F.softmax(logits, dim=-1)
            tok = torch.multinomial(p, 1)
            draft_tokens.append(tok)
            draft_probs.append(p.gather(-1, tok))
            cur = torch.cat([cur, tok], dim=1)

        # 2. target 一次 forward 所有候选位置
        full = torch.cat([out] + draft_tokens, dim=1)
        tgt_logits = target(full).logits[:, -(K+1):, :] / temperature
        tgt_probs  = F.softmax(tgt_logits, dim=-1)

        # 3. 拒绝采样
        accepted = 0
        for i in range(K):
            r = torch.rand(1, device=device)
            ratio = (tgt_probs[:, i, :].gather(-1, draft_tokens[i]) /
                     draft_probs[i].clamp(min=1e-12))
            if r < ratio.clamp(max=1.0):
                out = torch.cat([out, draft_tokens[i]], dim=1)
                accepted += 1
            else:
                # 拒绝,从修正分布采样一个 token 替代
                adj = (tgt_probs[:, i, :] - draft_probs[i] *
                       tgt_probs[:, i, :].new_ones(1, tgt_probs.size(-1))).clamp(min=0)
                adj = adj / adj.sum(-1, keepdim=True).clamp(min=1e-12)
                tok = torch.multinomial(adj, 1)
                out = torch.cat([out, tok], dim=1)
                break
        else:
            # K 个全接受,再取 target 在最后一个位置的采样作 bonus
            tok = torch.multinomial(tgt_probs[:, -1, :], 1)
            out = torch.cat([out, tok], dim=1)
        # print(f"accepted={accepted}/{K}")

    return out
```

> 生产实现请用 vLLM / TensorRT-LLM(内置 EAGLE / Medusa / MTP)。

## 🎨 21.10.A 通俗类比合集

### 类比一: 工厂提产线 🏭

> 推理 = 24 小时不停的工厂,优化 = 提产能。

- **Flash Attention** = 把"频繁去仓库取料"改成"工人桌上小托盘",省一半搬运时间
- **PagedAttention + Continuous Batching** = 来一单做一单,不再"凑满 10 单才开工"
- **Chunked Prefill** = 大订单拆小批次,与零散订单**交错**,大客户不挡其他客户
- **Speculative Decoding** = 实习生先快猜下一步,师傅一次性检查批准
- **量化** = 用同等精度的小一号扳手,工序更快
- **Compile** = 把多道工序合并成一道流水线工位

### 类比二: 餐厅外卖优化 🛵

> 厨房 = 大模型;骑手 = decode。

- **Speculative** = "预备出餐",外卖小哥拿"猜测订单"先送,客户确认就成,不确认就退回
- **Flash Attention** = 厨房布局优化,减少厨师跑动
- **Chunked Prefill** = 大订单分批出锅,不让单笔大单堵满后厨
- **Sliding Window** = "只看最近 5 张工单",别让历史挤爆告示牌
- **量化** = 不再全用一线食材,二三线大量替代,口感差异小但成本砍半

### 类比三: 高速公路改造 🚗

> 数据流是车流。

- **Flash Attention** = 把"普通公路 + 红绿灯"改成"高架直达",省红灯
- **PagedAttention** = 多车道 + 车位共享,不再固定车位
- **Continuous Batching** = 没有"必须凑齐 50 辆才能开"的规定,随到随走
- **Speculative** = 摩托车先行探路,大货车一次走稳路段
- **Sliding Window** = 不看 100 公里外,只看眼前 5 公里 → 决策快
- **量化** = 把 18 吨大货车换成 5 吨皮卡,运量略降但跑得飞快

### 三个类比对比

| 类比 | 强调什么 | 适合记忆什么 |
| --- | --- | --- |
| 🏭 工厂 | **工序合并 / 实习生** | Flash / Speculative |
| 🛵 外卖 | **预出餐 / 食材替代** | MTP / 量化 |
| 🚗 公路 | **直行 / 多车道** | PagedAttention / Continuous |

**记忆口诀**:
> **"Flash 砍 IO,Page 救显存,Chunk 稳节奏,Spec 借小模型,量化全面降本。"**

## 21.11 常见疑问

### Q1: Flash Attention 与普通 attention 数值一致吗?
**A**: 数学等价,但 fp16/bf16 下有微小数值差异(累加顺序不同 → 末位变动)。对 logits 影响 < 1e-3,对最终 token 选择几乎无差。

### Q2: Speculative Decoding 真的不掉质量吗?
**A**: 在标准拒绝采样的算法下,**输出分布严格等于直接对 target 采样**——数学保证不掉质量。但工程实现中若简化(如 lossy speculative)会引入微小差异。

### Q3: chunk size 怎么调?
**A**:
- TPOT 优先 → chunk_size = 256
- TTFT 优先 → chunk_size = 1024+
- 默认 vLLM 512 是合理起点

### Q4: AWQ vs GPTQ 选哪个?
**A**:
- 推理速度: 两者接近(同 4-bit kernel)
- 精度: AWQ 略稳(activation-aware),GPTQ 在 OBQ 系列上理论好
- 工程: AWQ 量化时间长,GPTQ 快
- 默认: 优先 AWQ(社区主流)

### Q5: torch.compile 在变长输入下会卡吗?
**A**: 是的——动态形状会触发重 trace。对 LLM 推理,主流做法是:
- 用 dynamic=False + 输入按 bucket padding
- 或用 vLLM 的自定义 CUDA kernel,不依赖 torch.compile

## 21.12 本章小结

### 核心优化族

```text
1. Flash Attention      → IO 减少,attention 2-5×
2. Paged + Continuous   → 吞吐 3-10×,显存 ×3 利用
3. Chunked Prefill      → TPOT 平稳
4. Speculative / MTP    → decode 吞吐 2-3×
5. Sliding Window       → 长上下文显存 O(n) → O(w)
6. Compile / TRT-LLM    → 整体 1.3-2×
7. 量化(W/KV/Act)      → 显存 / 速度叠加
```

### 核心要点

| 要点 | 关键 |
| --- | --- |
| **roofline** | 区分算力 / 带宽 bound |
| **Flash** | tiling + fused kernel,IO-aware |
| **Speculative** | 用小模型 / MTP 头并行猜 |
| **量化** | 多维(W/KV/Act)叠加 |

### 一句话总结

> **推理优化是多维度组合拳:Flash 砍 attention IO、PagedAttention + Continuous Batching 解锁吞吐、Chunked Prefill 让 TTFT 与 TPOT 不打架、Speculative / MTP 让 decode 一次跨多步、量化把权重 / KV / 激活同时压低——这套叠加可以让同一张 H100 服务量从 baseline 翻几倍,是 LLM 商业化的关键护栏。**
