## 摘要

模型训好只是第一步,**真正服务用户的瓶颈在推理**——延迟、吞吐、显存、并发,每一项都直接对应美元。本章先把"一次自回归生成"拆解到最细:

- **Prefill**:把 prompt 一次性 forward,建好 KV cache。算力受限,大 batch
- **Decode**:每次只生成 1 个 token,反复 forward,显存带宽受限,小 batch
- 两阶段算力 / 带宽特性截然不同,**不应平等对待**

由此延伸出现代推理系统的核心设计:Continuous Batching、Prefill / Decode 解耦、调度策略,以及评估指标 TTFT / TPOT。

读完你能解释为什么 vLLM、SGLang、DeepSeek-V3 推理系统都把 Prefill 与 Decode 当两种不同任务来调度。

## 18.1 推理 vs 训练:本质差异

| 维度 | 训练(§13) | 推理 |
| --- | --- | --- |
| Forward | ✅ | ✅ |
| Backward | ✅ | ❌ |
| 优化器状态 | ≈ 12P bytes | 0 |
| 显存大头 | 激活 + Adam | **权重 + KV cache** |
| 单次 step 算力 | $6P \cdot T$ | $2P \cdot T$(prefill)/ $2P$(decode) |
| 主要瓶颈 | 算力 + 通信 | **显存带宽**(decode) |

**关键直觉**:推理时 decode 阶段每 step 只输出 1 个 token,但要把整个 model weight 从 HBM 搬到 SM 一次 → **算力远未饱和,带宽先到顶**。这是几乎所有推理优化的出发点。

## 18.2 自回归生成的两阶段:Prefill / Decode

### 18.2.1 Prefill 阶段

输入完整 prompt $x_1, \ldots, x_n$,**一次 forward**:

```text
1. 把 prompt 全部 tokenize → input_ids[1..n]
2. 一次性 forward,计算每一层每一位置的 K, V
3. 把 K, V 写入 KV cache(共 n×d_kv 字节/层)
4. 取最后位置的 logits,采样得到第 1 个新 token
```

**算力 = $2 P \cdot n$**(与 batch token 数线性)。

### 18.2.2 Decode 阶段

从上一步生成的 token 开始,**每次只输入 1 个 token**:

```text
for step in range(max_new_tokens):
    1. forward 1 个 token(用之前积累的 KV cache)
    2. 读 KV cache:每层都要把历史 n+step 个 token 的 K, V 读出来 → bandwidth heavy
    3. 算 attention:复杂度 O(n+step) 而非 O(n²)
    4. 把新 K, V append 到 cache
    5. 采样新 token
```

每 step 算力 ≈ $2 P$,但**显存带宽 ≈ 2P + 2·layers·(n+step)·d_kv** bytes。

### 18.2.3 两阶段的工程性差异

| 维度 | Prefill | Decode |
| --- | --- | --- |
| 每 step token 数 | n(整段) | 1 |
| 算力(per step) | $2P n$ | $2P$ |
| KV cache 读取 | 不读,只写 | **每层都读全部历史** |
| 受限于 | **算力** | **显存带宽** |
| Batch 友好度 | ✅ 大 batch 高吞吐 | 中等(每 token 占带宽) |
| 延迟敏感度 | 决定 TTFT | 决定 TPOT |

**这两个阶段几乎是两种不同的工作负载**,后面 §18.7 会看到工业界已经在物理上把它们分开。

## 18.3 单请求 forward 详解

### 18.3.1 推理时的张量形状

|阶段 | 输入 | KV cache | 输出 |
| --- | --- | --- | --- |
| Prefill | (1, n, d) | 写入 (layers, n, h_kv, d_k) | (1, n, V) → 取最后 1 个 |
| Decode step | (1, 1, d) | 读历史 + 追加 1 | (1, 1, V) |

注意:输入序列维度从 prefill 的 $n$ 变成 decode 的 $1$,这正是为什么 decode 算力极低、带宽极高。

### 18.3.2 RoPE 在推理时

RoPE(§06)与位置 $i$ 绑定。decode 时:

- 主流实现把 **prefill 已计算的 K 在写入 cache 时就完成旋转**;decode 当前 step 的 K / Q 用当前 offset 旋转
- 这样 cache 里的 K 是"已位置编码版",直接读出就能与新 Q 做内积
- 注意:扩展上下文(YaRN)修改 RoPE 频率会让 cache 中已旋转 K **作废**(§06.10)

### 18.3.3 停止条件

推理 loop 何时停?

- 达到 max_new_tokens
- 输出 EOS / 特殊 stop token
- 用户传入 stop string(检测 token 序列出现)
- 超时 / 用户中断

工程上还要小心 token 级 stop string 检测(`</s>` 可能被拆成多 token,逐 token 比对失败)。

## 18.4 Batch 推理与 Padding 困境

### 18.4.1 朴素 batch

把 N 个请求的输入用 padding 对齐到同一长度:

```text
req1: [Hi, how, are, you,  ?, PAD,  PAD,  PAD ]
req2: [The, sun, sets, in, the, west, today, .]
```

forward 一次,所有请求并行。

### 18.4.2 padding 浪费

- Prefill:padding token 浪费算力(占 attention 但无意义)
- Decode:**每个请求长度不同**,某请求生成完 EOS 后仍要陪跑等其他请求结束 → 严重浪费

实测:朴素 batch 在长尾分布下 GPU 利用率常 < 30%。

### 18.4.3 静态 batch 的两条妥协

- **prefill batch + decode batch 分离**:prefill 段大 batch,decode 段重新打包
- **按长度排序后分桶**(length bucketing):同桶内 padding 浪费小

仍不够好 → 引入 Continuous Batching。

---

## 18.5 Continuous Batching(连续批处理)

### 18.5.1 思路

不再"等同 batch 所有请求结束",而是**每个 step 动态加入新请求 / 移除已完成请求**。

```text
step 1: [req_A(decode), req_B(decode), req_C(prefill)]
step 2: [req_A(decode), req_B(done!), req_C(decode), req_D(prefill)]   ← B 完了换成 D
step 3: [req_A(decode), req_C(decode), req_D(decode), req_E(prefill)]
```

每个 step 是一次混合 forward——既包含某些请求的 decode,也可能包含新请求的 prefill。

### 18.5.2 显存视角:KV cache 池化

支持这种动态需要 KV cache 不再绑定固定 slot,而是**按页(page)分配**(详见 §19 PagedAttention)。这是 vLLM 的核心创新。

### 18.5.3 收益

- 吞吐 2-10×(相比静态 batch)
- 单 GPU 上可同时服务几十-上百并发请求
- 等待新请求加入的延迟极小

### 18.5.4 关键约束

- **Chunked prefill**:长 prompt 不能一次塞进 batch,否则挤压同 step decode 请求 → 分块 prefill,与 decode 交错(详见 §21)
- **公平性**:长请求不能饿死短请求,要调度策略支持

## 18.6 调度策略:FCFS / SJF / 优先级

推理引擎里的 scheduler 决定:**下一 step 把哪些请求放进 batch**。

### 18.6.1 FCFS(先到先服务)

简单公平,但长 prompt 阻塞短请求 → tail latency 差。

### 18.6.2 SJF / SRJF(短作业 / 短剩余优先)

预估剩余长度,短的优先 → 平均延迟最小。代价:可能让长请求饿死。

### 18.6.3 优先级队列

按用户级别 / SLA 优先级分组,高优先级请求抢占资源。商业部署常用。

### 18.6.4 Preemption(抢占)

KV cache 满时,**驱逐**某些低优先级请求的 cache,腾位给新请求。被驱逐的请求**重新计算 prefill**——代价高但避免 OOM。vLLM 默认支持。

### 18.6.5 实战:vLLM 默认 ≈ FCFS + 公平 quota + 抢占式回收;SGLang 增加更多 prefix cache 感知调度。

## 18.7 Prefill / Decode 解耦(Disaggregated Serving)

### 18.7.1 动机

回到 §18.2.3 表:Prefill 算力受限,Decode 带宽受限。**同一张 GPU 上混跑会互相干扰**:

- Decode batch 加进一个长 prompt 的 prefill → 该 step 时间 5-10× 暴增 → TPOT 抖动剧烈

### 18.7.2 解耦方案

把请求分成两阶段、由两组 GPU 处理:

```text
请求 → [Prefill 集群:大模型计算 → 产出 KV cache] → 网络传输 KV → [Decode 集群:增量生成]
```

- Prefill 集群:用算力强的卡(H100),大 batch
- Decode 集群:用 KV cache + 高带宽 HBM(也可以是相同卡,但参数不同)
- 两类资源各自最优配置,**TTFT 与 TPOT 互不干扰**

### 18.7.3 代表实现

- **DistServe**(SOSP 2024)、**Mooncake**(Kimi 论文)、**Splitwise**(MSR)
- **DeepSeek 推理栈**(2024-2025)采用解耦 + 跨节点 KV 传输 + EP(MoE)的极致工程

### 18.7.4 代价

- **KV cache 跨网络传输**:层数 × n × d_kv 字节,对 70B+ 模型上百 MB 级,需要 RDMA / InfiniBand
- 调度系统复杂(prefill / decode 队列、KV 路由)

只在**大流量场景**值得;小规模直接 vLLM/SGLang 单引擎就够。

## 18.8 关键性能指标:TTFT / TPOT / Throughput

### 18.8.1 三大指标

| 指标 | 全称 | 含义 | 用户体感 |
| --- | --- | --- | --- |
| **TTFT** | Time To First Token | 请求到达 → 第 1 个 token 出 | 多久开始"打字" |
| **TPOT** | Time Per Output Token | 后续每个 token 间隔 | "打字速度"(字/秒) |
| **Throughput** | tokens / sec / GPU | 集群总输出 | 美元成本主指标 |

### 18.8.2 三者矛盾

- 大 batch → throughput ↑、TPOT ↑(变慢)、TTFT ↑(prefill 排队)
- 小 batch → 反向

工业部署常按 SLA 设定:p99 TTFT < 500ms,p99 TPOT < 50ms,在此约束下最大化 throughput。

### 18.8.3 输出速度的人类感知

- < 100 tokens/s:略卡顿
- 100-300 tokens/s:流畅
- 500+ tokens/s:快到看不过来,可省略 streaming

人眼阅读速度约 150 wpm ≈ 200-300 tokens/s 已足够。

## 18.9 完整代码:可运行的 KV-cache 推理

下面把 §10.6 的 GPTMini 扩成 KV-cache 推理(简化版,真实生产用 vLLM):

```python
import torch, torch.nn.functional as F

@torch.no_grad()
def generate(model, prompt_ids: torch.Tensor, max_new_tokens: int,
             temperature: float = 1.0, top_k: int | None = None):
    """
    prompt_ids: (1, n) long
    返回 (1, n + max_new_tokens) long
    """
    model.eval()
    device = prompt_ids.device
    past_kvs = [None] * model.n_layers     # 每层 (K, V)

    # === Prefill ===
    logits, past_kvs = model.forward(prompt_ids, past_kvs)  # forward 支持 KV cache
    last_logits = logits[:, -1, :]
    next_tok = _sample(last_logits, temperature, top_k)
    out = torch.cat([prompt_ids, next_tok], dim=1)

    # === Decode ===
    for _ in range(max_new_tokens - 1):
        logits, past_kvs = model.forward(next_tok, past_kvs)
        last_logits = logits[:, -1, :]
        next_tok = _sample(last_logits, temperature, top_k)
        out = torch.cat([out, next_tok], dim=1)
        if next_tok.item() == model.eos_id:
            break
    return out

def _sample(logits, temperature, top_k):
    logits = logits / max(temperature, 1e-5)
    if top_k:
        v, _ = torch.topk(logits, top_k)
        logits[logits < v[:, [-1]]] = -float("inf")
    probs = F.softmax(logits, dim=-1)
    return torch.multinomial(probs, 1)
```

`model.forward(x, past_kvs)` 内部:

- 若 `past_kvs[l]` 为 None → prefill 模式,所有位置计算 + 缓存
- 否则 → decode 模式,只对新 token 计算 Q,K,V,与 cache concat 后做 attention

## 🎨 18.10.A 通俗类比合集

### 类比一: 餐厅:点餐 vs 端菜 🍽️

> 一次自回归生成像顾客点完餐 → 一道一道上菜。

- **Prefill** = 服务员**记完整道菜单**(把整段 prompt 读进来),写好工单
- **Decode** = 每隔几秒上一道菜,逐道直到吃完
- **KV cache** = 后厨已经备好的高汤,新菜随取随用
- **Padding** = 跟一桌人一起点餐,有人吃完了但仍要等其他人——浪费餐位
- **Continuous Batching** = 新顾客随时入座、老顾客吃完就走,**没有"开桌等齐"**
- **TTFT** = 你坐下到第一道菜上桌的时间(开局慢=体验差)
- **TPOT** = 一道道菜之间的间隔(节奏)

### 类比二: 直播打字员 ⌨️

> Chat 模型像一个直播打字员:你抛一个问题,他从头打到尾。

- **Prefill** = 打字员先**通读你的问题**(prompt),理解上下文
- **Decode** = 然后**逐字打输出**,每个字看一下之前自己打过的全文
- **KV cache** = 他大脑里"已经想好的上下文索引",不必每打一字都重读
- **Padding 浪费** = 让 10 个打字员各自写不同长度的稿子,慢的还没写完,快的只能干等
- **Continuous Batching** = 来一个开一个,完了下一个,大家在一个房间互不耽误
- **解耦** = 一组打字员专管"读题",另一组专管"码字",流水线分工

### 类比三: 烧水 vs 倒水 🫖

> Prefill 是烧水,Decode 是倒水。

- **烧水(prefill)** = 一次性烧一壶,**算力受限**,小心别让水流不畅
- **倒水(decode)** = 一杯一杯倒,**带宽受限**(壶嘴粗细)
- **KV cache** = 热水瓶,烧好的水保温,后续直接倒
- **Continuous Batching** = 边烧边倒,新客人来直接倒,水快倒完时再烧一壶
- **解耦** = 一边专烧水,一边专倒水,互不打扰

### 三个类比对比

| 类比 | 强调什么 | 适合记忆什么 |
| --- | --- | --- |
| 🍽️ 餐厅 | **两阶段服务** | Prefill / Decode |
| ⌨️ 直播打字 | **逐字生成** | KV cache 的"已想好" |
| 🫖 烧水 | **算力 vs 带宽** | 解耦的根本动机 |

**记忆口诀**:
> **"Prefill 一锅烧,Decode 一杯倒;KV 是热水瓶,padding 是浪费,Continuous 一直供。"**

## 18.11 常见疑问

### Q1: 为什么 decode 比 prefill 慢得多(per token)?
**A**: 看似奇怪——decode 算力少。但每 step 都要把整个模型权重从 HBM 搬一次(2P bytes)+ 读 KV cache,**带宽受限**。所以单 token 延迟 = 模型大小 / HBM 带宽,与算力关系小。这就是为什么"decode 是 memory-bound"。

### Q2: KV cache 能不能不存,每次重算?
**A**: 不能。每次 decode 重算 = 复杂度 $O(n^2)$ 退化为 $O(n^3)$ 总成本。对于 4k 上下文,1000 步 decode 重算会比有 cache 慢 100×+。

### Q3: vLLM 的 Continuous Batching 有什么前提?
**A**: 需要 PagedAttention 让 KV cache 不强制连续(§19)。如果直接用 nn.MultiheadAttention 默认实现是不行的。

### Q4: TTFT 太长怎么办?
**A**:
- prompt 太长 → chunked prefill(§21)
- 排队 → 提升 prefill 集群算力或解耦部署
- 模型太大 → 量化(§22)

### Q5: 在线服务的 batch_size 越大越好吗?
**A**: 不是。大 batch 提升 throughput,但增加 TTFT 与 TPOT 抖动。SLA 约束下找平衡:vLLM 默认 `max_num_seqs = 256` 是 H100 上的甜点。

## 18.12 本章小结

### 核心模型

```text
请求到达 → Prefill(算力) → 写 KV cache
       → Decode(带宽) → 逐 token,读+追加 cache → EOS / max → 完成
```

### 核心要点

| 要点 | 关键 |
| --- | --- |
| **Prefill** | 大 batch、算力受限 |
| **Decode** | 小 batch、带宽受限 |
| **Continuous Batching** | 静态 batch 的替代品 |
| **解耦部署** | 大流量场景 |
| **指标** | TTFT / TPOT / Throughput,三者矛盾 |

### 一句话总结

> **LLM 推理是"两个性格相反的阶段拼在一起":Prefill 一次吃整段 prompt(算力受限),Decode 逐 token 喷出(带宽受限);现代推理系统通过 KV cache + Continuous Batching 把吞吐提上来,大流量再用 Prefill/Decode 解耦把 TTFT 与 TPOT 互不干扰地各自最优。**
