## 摘要

模型输出 logits 后,**怎么把 V 维实数变成一个 token** —— 这是"采样"的核心问题。本章按演进顺序梳理:

- **Greedy / 温度 / 多项式**:三种基础姿势
- **Top-k / Top-p / Min-p**:截尾分布的三种方法
- **重复惩罚 / DRY**:防止"复读机"
- **Beam Search**:翻译时代主力,LLM 时代式微
- **结构化输出**:让模型必产合法 JSON / 正则
- **Logit bias / processor**:在 logits 上挂任意逻辑

读完你能解释 OpenAI、Anthropic、vLLM 提供的所有 sampling 参数,并按场景写出"配方"。

## 20.1 为什么要采样而不是 argmax?

### 20.1.1 Greedy 的两大弊

直接取 argmax 简单,但:

- **重复死循环**:模型很容易陷入"... I ... I ... I ..."的局部最优
- **缺乏创造性**:同一 prompt 永远同一输出
- **长尾不可达**:正确答案可能是排名第 2、3 但合理的 token

### 20.1.2 采样的两条意义

1. **多样性**:同一 prompt 可生成不同回答(对话、创意写作场景)
2. **打破局部最优**:跳出陷阱,有时反而更好

### 20.1.3 何时反而 greedy?

- 代码补全、数学证明、API 工具调用:**确定性最优**
- evaluation benchmark:可重复
- 蒸馏:用作 teacher 生成

## 20.2 三种基础采样:Greedy / Temperature / Multinomial

### 20.2.1 Greedy(argmax)

$$
y = \arg\max_i z_i \tag{20.1}
$$

简单确定。

### 20.2.2 温度采样

按 softmax 概率多项式采样,温度 $\tau$ 调形:

$$
p_i = \frac{\exp(z_i / \tau)}{\sum_j \exp(z_j / \tau)} \tag{20.2}
$$

- $\tau \to 0$:接近 argmax
- $\tau = 1$:原始 softmax
- $\tau > 1$:分布更平,更随机
- $\tau \to \infty$:近似均匀分布

### 20.2.3 默认多项式采样

直接按 $p$ 采样,无任何截断 → 容易出现"长尾噪声 token"被选中,生成质量不稳。所以工业上都配合 Top-k / Top-p。

---

## 20.3 截断采样:Top-k / Top-p / Min-p

### 20.3.1 Top-k(Fan et al., 2018)

只保留概率最大的 $k$ 个 token,重新归一化后采样:

$$
\tilde p_i = \begin{cases} p_i / Z & i \in \text{top-}k \\ 0 & \text{else} \end{cases} \tag{20.3}
$$

$k$ 典型 40-100。问题:**固定 $k$ 不适应分布形态**。分布尖时多取了噪声,分布平时又截太多。

### 20.3.2 Top-p / Nucleus(Holtzman et al., 2020)

按概率降序累加,取累加到 $p$ 的最小集合:

$$
\mathcal{V}_p = \arg\min_{S} |S| \text{ s.t. } \sum_{i \in S} p_i \geq p \tag{20.4}
$$

$p = 0.9$ 是常见值。自适应**:分布尖时只取 1-2 个,分布平时取很多个。

### 20.3.3 Min-p

近年新派(LLaMA-3 / Mistral 默认推荐):**只保留概率 ≥ p_min × max(p) 的 token**:

$$
\mathcal{V}_{\min} = \{ i : p_i \geq p_{\min} \cdot \max_j p_j \} \tag{20.5}
$$

$p_{\min} = 0.05$ 是经验值。优势:**完全按相对概率截断**,更稳健。

### 20.3.4 配合温度

工程上通常**先温度,再截断**:

```text
1. logits / τ
2. softmax → p
3. Top-k / Top-p / Min-p 截断
4. 重新归一化
5. 多项式采样
```

也可"先截断再温度",效果略不同。多数实现按 1-5 顺序。

## 20.4 重复惩罚:Repetition / Frequency / Presence / DRY

### 20.4.1 现象

不加惩罚时模型容易**复读**,尤其在长输出 / 小模型 / 温度低时:

```text
The cat sat on the mat. The cat sat on the mat. The cat sat on the mat...
```

### 20.4.2 Repetition Penalty(Keskar et al., 2019)

已出现过的 token 的 logit 除以 $\rho > 1$:

$$
z_i \leftarrow \begin{cases} z_i / \rho & \text{if } z_i > 0 \\ z_i \cdot \rho & \text{if } z_i < 0 \end{cases}, \quad i \in \text{history} \tag{20.6}
$$

$\rho \approx 1.1\text{-}1.3$。简单但对短重复无力(不是 token 级而是"短语级"重复)。

### 20.4.3 Frequency / Presence Penalty(OpenAI 风格)

直接减一个偏置:

$$
z_i \leftarrow z_i - \alpha \cdot \text{count}(i) - \beta \cdot \mathbb{1}[\text{count}(i) > 0]
$$

- frequency:与出现次数成比例
- presence:只要出现过都扣固定值

$\alpha \in [0, 2]$, $\beta \in [0, 2]$。

### 20.4.4 DRY(Don't Repeat Yourself)

最新派(text-generation-webui / EXL2 等):**检测最近 $n$-gram 重复模式**,把那些"会延续重复"的 token logit 大幅降低。

- 直接打击"短语级复读"
- 不影响合法重复(例:列表里多个"-")

### 20.4.5 实战

- chat / 写作:repetition_penalty = 1.05-1.1
- 长输出:开 DRY,$\alpha = 0.8$
- 代码:0(代码本就有大量合法重复)

## 20.5 Beam Search 与 Best-of-N

### 20.5.1 Beam Search

不只跟踪一个候选,而是同时维护 $B$ 个**部分序列**,每步保留累积 log-prob 最高的 $B$ 个延伸。

机器翻译时代主力,LLM 时代用得少:

- **多样性差**:beam 互相相似
- **长输出退化**:倾向短而高概率
- **算力 ×B**

### 20.5.2 Best-of-N(BoN)

独立采样 $N$ 个完整候选,用一个 reward model / 验证器选最优。

- 不影响 sampler 本身
- 用于 reasoning / 测评(Self-Consistency)
- 算力 ×N,但天然支持任意采样策略

### 20.5.3 现状

- chat / 流式:几乎全用普通 sampler(Greedy 或 Temperature + Top-p)
- reasoning / 数学:Best-of-N 或 majority vote
- 翻译 / 摘要:仍可见 Beam Search

## 20.6 受限解码:JSON Mode / 正则 / 语法

### 20.6.1 问题

让模型输出**合法 JSON / SQL / 特定正则**——只靠 prompt 不够稳定。

### 20.6.2 思路:在 sampler 层屏蔽非法 token

每步:

1. 模型输出 logits
2. **遍历 vocab,把"如果选此 token 会导致违法"的 logit 设为 $-\infty$**
3. 在剩余合法 token 中正常采样

只要语法可形式化(JSON、CFG、正则),就能保证 100% 合法。

### 20.6.3 主流实现

- **Outlines** / **lm-format-enforcer**:基于 FSA(有限状态自动机)
- **vLLM**:内置 grammar-based decoding
- **OpenAI Structured Output**(2024 后):服务端原生支持 JSON schema

### 20.6.4 性能代价

- 每步多一次"vocab 过滤" → ~5-15% 延迟开销
- 复杂语法树构造开销更大,需要预编译

收益:让弱模型也能稳出 JSON,工业 agent 必备。

## 20.7 Logit Bias 与 Logits Processor

### 20.7.1 Logit Bias

对指定 token 加偏置:

```python
logits[token_id] += bias   # bias > 0 鼓励;< 0 抑制
```

OpenAI API 接收 `logit_bias: {token_id: ±100}` 字典。

应用:

- 禁词:`bias = -100`
- 强制带某词:`bias = +100`
- 调风格:增加某些词的概率

### 20.7.2 Logits Processor 流水线

HuggingFace、vLLM 都支持"logits processor"接口——把任意函数串成流水线:

```text
logits → temperature → top_p → repetition_penalty → custom_grammar → 采样
```

每一步都是"输入 logits → 输出 logits"。这是把所有 sampling 技巧统一的工程抽象。

### 20.7.3 写一个自定义 processor

```python
class BanWordProcessor:
    def __init__(self, banned_token_ids):
        self.banned = banned_token_ids
    def __call__(self, input_ids, scores):
        scores[..., self.banned] = -float("inf")
        return scores
```

挂在 `generate(logits_processor=[...])` 即可。

## 20.8 Stop 与 EOS 的工程坑

### 20.8.1 EOS

模型自然学会输出 EOS(end-of-sequence)。检测到就停。每个模型 EOS id 不同 → 不能写死。

### 20.8.2 多 EOS

LLaMA-3 / Qwen 等有**多个停止 token**(`<|im_end|>`、`</s>` 等)。`tokenizer.eos_token` 可能只是其中一个,要看 model_card 的 `eos_token_ids` 列表。

### 20.8.3 Stop String

用户指定的 "停止字符串"(如 `"\n\n"`)需要在 token 层匹配。陷阱:

- 一个字符串可能被 tokenize 成多个 token
- 同字符串在不同上下文 token 化结果可能不同
- 正确做法:**累积 detokenize 后字符串匹配**,而不是 token 比对

### 20.8.4 max_tokens

硬上限。安全网,防止跑飞。注意"max_tokens"通常指**新生成的 token 数**,不含 prompt。

## 20.9 完整代码:统一 sampler

```python
import torch
import torch.nn.functional as F

def sample(
    logits: torch.Tensor,            # (B, V)
    temperature: float = 1.0,
    top_k: int | None = None,
    top_p: float | None = None,
    min_p: float | None = None,
    repetition_penalty: float = 1.0,
    history_ids: torch.Tensor | None = None,
    logit_bias: dict[int, float] | None = None,
):
    logits = logits.clone()

    # 1. logit_bias
    if logit_bias:
        for tid, b in logit_bias.items():
            logits[..., tid] += b

    # 2. repetition penalty
    if repetition_penalty != 1.0 and history_ids is not None:
        gathered = logits.gather(-1, history_ids)
        gathered = torch.where(gathered > 0, gathered / repetition_penalty,
                                              gathered * repetition_penalty)
        logits.scatter_(-1, history_ids, gathered)

    # 3. temperature
    if temperature == 0.0:
        return logits.argmax(-1, keepdim=True)
    logits = logits / temperature

    probs = F.softmax(logits, dim=-1)

    # 4. min-p
    if min_p:
        max_p = probs.max(-1, keepdim=True).values
        probs = torch.where(probs >= min_p * max_p, probs, torch.zeros_like(probs))

    # 5. top-k
    if top_k:
        topk_vals, _ = torch.topk(probs, top_k)
        probs = torch.where(probs >= topk_vals[..., [-1]], probs, torch.zeros_like(probs))

    # 6. top-p
    if top_p:
        sorted_p, sorted_idx = torch.sort(probs, descending=True, dim=-1)
        cumsum = sorted_p.cumsum(-1)
        mask = cumsum > top_p
        mask[..., 1:] = mask[..., :-1].clone()
        mask[..., 0] = False
        sorted_p = sorted_p.masked_fill(mask, 0.0)
        probs = torch.zeros_like(probs).scatter_(-1, sorted_idx, sorted_p)

    # 7. 归一并多项式采样
    probs = probs / probs.sum(-1, keepdim=True).clamp(min=1e-12)
    return torch.multinomial(probs, 1)
```

每步可单独测试。

| 场景 | temperature | top_p | top_k | min_p | rep_pen | 备注 |
| --- | --- | --- | --- | --- | --- | --- |
| 代码补全 | 0.0-0.2 | 1.0 | — | — | 1.0 | greedy 最稳 |
| 数学 / 推理(单次) | 0.0-0.3 | 1.0 | — | — | 1.0 | 配 Self-Consistency 提温 |

## 🎨 20.10.A 通俗类比合集

### 类比一: 餐厅点餐 🍴

> 模型每步在 V 道菜里点一道。

- **Greedy** = 每次都点点击率第一的招牌菜 → 几乎每次同一个
- **Temperature** = 调整你的"探索心情":τ=0 老老实实,τ=2 闭着眼随便
- **Top-k** = 服务员说"今天前 40 道热门可选,其他暂停"
- **Top-p** = 服务员说"前几道菜占了 90% 销量,你只能从中选"
- **Min-p** = 服务员说"任何菜如果销量低于招牌菜的 5%,直接划掉"
- **Repetition penalty** = 你已经点过的菜,服务员主动建议换换口味
- **Beam search** = 同时点 4 桌不同搭配,最后选总分最高的桌

### 类比二: 抽奖机 🎰

> 每个 token 是一张奖券,概率不同。

- **Greedy** = 不抽奖,直接拿一等奖
- **Multinomial** = 真抽奖,但有时中安慰奖(噪声 token)
- **Top-k / Top-p** = "只把奖券前几张放进抽奖箱",避开渣奖
- **温度** = 抽奖箱摇得多猛(摇得猛 → 概率拉平)
- **JSON 受限解码** = "今天只抽以 `{` 开头的奖券,其他全扔"
- **DRY** = 上一轮已抽中的搭配,这一轮不能再出

### 类比三: 写作老师选词 ✍️

> 学生(模型)在写作文,每个词由"内心词典"打分。

- **Greedy** = 永远选脑中第一个想到的词 → 容易呆板
- **Top-p 0.9** = 候选词加起来要占 90% 信心,**少而精**
- **Min-p 0.05** = "只考虑信心至少是冠军 5% 的候选"
- **Temperature** = 老师叮嘱"今天放飞一点 / 今天严谨一点"
- **Repetition penalty** = "前文已经用过的词,这次少用"
- **Beam search** = 同时构思 4 段,最后选最连贯的一段交卷

### 三个类比对比

| 类比 | 强调什么 | 适合记忆什么 |
| --- | --- | --- |
| 🍴 餐厅 | **过滤候选** | Top-k / Top-p / Min-p |
| 🎰 抽奖 | **随机性控制** | Greedy / Temperature |
| ✍️ 写作 | **风格平衡** | 重复惩罚 / DRY |

**记忆口诀**:
> **"先温度后截断,Top-p 自适应,Min-p 看相对;重复惩罚防复读,JSON 受限保格式。"**

## 20.11 常见疑问

### Q1: temperature=0 等于 argmax 吗?
**A**: 数学上 $\tau \to 0$ 极限是 argmax。工程上多数实现把 `temperature=0` 直接走 greedy 分支,避免 NaN / overflow。

### Q2: Top-p 与 Top-k 该用哪个?
**A**: 优先 Top-p(自适应)。Top-k 主要在"硬截尾"场景(避免长尾噪声)。Min-p 是 Top-p 的稳健改进,新模型推荐。

### Q3: 我的模型输出总复读,怎么救?
**A**:
1. repetition_penalty = 1.1-1.3
2. 开 DRY
3. 提温度(0.7-1.0)
4. 仍不行 → 模型本身欠拟合或 SFT 数据有问题

### Q4: 我要稳定的 JSON 输出怎么办?
**A**:
1. prompt 给清晰 schema
2. 用 Outlines / lm-format-enforcer 或服务端 structured output
3. 配 temperature ≈ 0
4. 验证失败时重采,而非加大 retry

### Q5: Beam search 在 LLM 里真没用了?
**A**: 几乎。Translation / 摘要还能见;chat / 推理几乎全被 Top-p / Best-of-N 替代。原因:beam 输出多样性差、长输出退化、计算贵。

## 20.12 本章小结

### 核心流程

```text
logits → logit_bias → repetition_penalty → / temperature
       → softmax → min-p → top-k → top-p → 归一 → multinomial
```

### 核心要点

| 要点 | 关键 |
| --- | --- |
| **温度** | 控制确定 vs 多样 |
| **Top-p / Min-p** | 自适应截断,主流 |
| **Repetition / DRY** | 防复读 |
| **JSON 受限解码** | Agent 必备 |
| **场景配方** | 代码 0.0 / 对话 0.7 / 创意 1.0+ |

### 一句话总结

> **采样把 logits 翻译成 token:先温度软化、再截尾(Top-p / Min-p)、再防复读(repetition / DRY)、最后多项式抽样;结构化输出场景再叠 grammar mask 保格式。一套统一 logits processor 流水线可以承载所有变体,工业 LLM 服务的"采样面板"就是这条流水线的暴露。**
